## 총평 (Stone Face Mode)

> **"엔지니어링 감각은 상위권인데, '증명 하나를 끝까지 밀어붙인 흔적'이 없어 합격선 문턱에서 미끄러지는 전형적인 포트폴리오."**

v3 대비 구조·톤·tagline은 크게 개선됐다. 하지만 **치명적 약점 3개** — ①실측 수치 0건 ②비즈니스 공란 ③코드-발표 정합성 구멍 — 때문에 면접관은 "뽑고 싶은데 한 번 더 검증하자"로 기울게 된다. 수정 시 합격 확률 크게 상승.

**현재 상태 추정**: 상위 20% (이전 v3와 동일 평가). **P0 실행 후**: 상위 10% 합격권. **판정을 바꾸는 한 방**은 여전히 **실측 로그 1장**.

## Part 1. 강점 (유지할 것)

1. **Tagline이 선명·일관** — "모델 가속이 아니라 호출 구조 재설계". 15분 후에도 기억에 남음. v3의 최대 결함이 해결된 부분.
2. **핵심 통찰 내러티브 (섹션 3)** — 540회 계산 → 질문 전환. 면접관이 메모하게 만드는 구간.
3. **계층형 판정 + 측정 인프라가 실재** — 코드 검증 결과 `_StepTimer`(`pipeline.py:20-42`), 3단계 토픽 감지(`analysis/topic.py:1-110`), 6종 트리거(`analysis/triggers.py:43-173`) 모두 주장대로 구현됨. 엔지니어링 성숙도 시그널 충분.
4. **자료 수집 상태 게이트** (`positions ≥ 2 && consensus == null`)가 실제 코드(`pipeline.py:356`)와 정확히 일치. 도메인 구조 관찰이 코드로 내려온 사례로 강한 어필 포인트.

## Part 3. 치명적 약점 7가지 (v4 기준)

### 🔴 약점 1. 실측 수치가 하나도 완성되지 않았다 (최대 약점)

발표 전반이 "설계/가설/구조" 위주:
- 경로 분포 → "재구성 중"
- F1 → "다음 마일스톤"
- 400ms → "목표"
- 1/5 호출 절감 → "예상"

**면접관 속마음**: *"좋은 감각은 있는데 뭐 하나 끝까지 한 게 있나?"*

부록 A **Q10**("다른 후보는 F1=0.87을 들고 왔다")을 스스로 예상하고도 **실측 하나를 안 만든 채** "스코프 선택"으로 방어하는 건 약함. 채용은 스코프 논쟁이 아니라 **완성도 경쟁**임.

**로그 파일은 존재하지만**(`logs/server.log`, 623KB) 경로 A/B/C 분포 재구성 스크립트가 없어 "데이터는 있는데 숫자는 못 뽑았다" 상태. 이건 치명적.

### 🔴 약점 2. 코드-발표 정합성 구멍 (적대적 면접에서 위험)

**구멍 1 — 원칙 3(비동기 디커플링)**
- 발표 L193-199에서 "`asyncio.create_task`로 LLM 백그라운드"라 단언
- 실제: `api/websocket.py:107`에 **주석만 존재**, 실제 호출 없음
- 파이프라인은 제공자별 전략(Ollama 순차 / OpenRouter 병렬)으로 구현됨
- **면접관이 "해당 코드 보여주세요" 한 번이면 원칙 3 전체가 흔들림**

**구멍 2 — JSON 스키마 강제 (부록 A Q4)**
- L334 "JSON 스키마 강제 (pydantic 검증)"
- 실제: Pydantic 없음. `_merge_positions()` 수동 후처리만
- **재시도 로직도 없음** (`llm.py:128-135`는 예외 처리만)

**구멍 3 — VAD**
- L290, L371 "Silero 통합 준비, 현재 RMS로 동작"
- 실제: sherpa-onnx SileroVAD 설정(`stt/__init__.py:93-96`)과 RMS 로직(`stt/whisper_stt.py:40-49`)이 **둘 다 존재**
- "지금 실제 활성 경로는 뭔가요?" 질문 받으면 즉답 어려움

### 🔴 약점 3. 비즈니스 섹션이 여전히 공란

v3 리뷰 후 섹션 2(비즈니스 맥락)가 추가됐지만, 내용이 얇다:
- "법무·M&A·이사회·인사" → **나열만**
- 단 하나의 **수치·고객 사례·검증**이 없음
- "로컬이어야만 쓸 수 있다"는 멋진 문장이지만, **누구에게 확인한 얘기인가?** 가정인지 검증인지 불명확

비개발자 면접관(대부분 사업부/HR)이 가장 점수를 깎는 구간. **"이거 팔리는가?" 질문에 대한 답이 없음.**

---

### 🔴 약점 4. 원칙 3(비동기)의 대칭성이 인위적

부록 A **Q1**에서 스스로 "순서/타이밍/병렬성"으로 구조화한 게 오히려 **"억지로 3개 만든 느낌"**을 강화.

특히 원칙 3은:
- **코드 구현이 얇음** (약점 2 참조)
- 개념적으로도 "비동기 UI"는 너무 일반적이라 독창성 약함
- 없어도 원칙 1·2만으로 발표의 중심 논지는 유지됨

→ **대칭 깨고 "2축(원칙 1+2) + 메타 원칙(측정 인프라)"** 구조가 더 정직하고 강함.

---

### 🔴 약점 5. 오프닝이 여전히 조건부

L26: "30초 데모 재생 **or** UI 스크린샷 1장" — **or** 이 치명적.
- 발표 첫 30초가 전체 인상의 60%
- 스크린샷 1장만으로는 15분 동안 "이게 뭐였더라" 반복됨
- 데모 **강제**로 바꿔야 함

또한 tagline을 **청중이 문제를 모르는 상태**에서 오프닝에 먼저 던지면 추상적으로 들림. "문제 → 시도 → 좌절 → 질문 전환" 순서로 가는 게 극적 효과 훨씬 큼.

---

### 🔴 약점 6. 부록 Q&A 10개는 인지 과부하

사람이 긴장 상태에서 자연스럽게 꺼낼 수 있는 스크립트는 **4-5개**가 한계. 10개 외우려 하면 **예상 밖 질문에 흔들림**.

- Q1, Q3, Q6 같은 자연스러운 질문은 본문 내용으로 답 가능 → 스크립트 불필요
- Q7, Q9, Q10(킬러) 중심 **5개로 압축**

→ LLM 실호출: 16회 (단순 발화당 3회 가정 441회 대비 3.6%)

### 파이프라인 디버깅 — 실측 과정에서 발견·수정한 5가지

이 과정 자체가 면접에서 **"실제로 시스템을 돌려봤다"** 는 증거. Q&A에 활용 가능.

1. **`routes.py:111` — 존재하지 않는 `WhisperFileSTT` import**
   → `WhisperSTT`로 수정. 배치 업로드 경로가 그동안 한 번도 실제로 동작한 적 없었음을 확인.
2. **`mlx-whisper` 미설치** — venv의 의존성 누락 (Apple Silicon + Python 3.14 + mlx-whisper 0.4.3 조합으로 해결)

### 측정에서 드러난 한계 (발표에 정직히 포함 권장)

1. **silence 트리거 0건의 원인** — 배치 업로드는 utterance가 한꺼번에 주입되므로 "발화 간 침묵" 감지 로직이 작동하지 않음. 이 트리거는 **실시간 WebSocket 경로에서만 의미**. → 발표 시 "배치 업로드는 STT·분석 검증용, 실시간 경로는 WebSocket 스트림에서 별도 작동" 문장으로 명시.
2. **화자 분리 TTS 특성에 민감** — sherpa-onnx 3dspeaker 임베딩이 edge-tts 합성 음성의 미세 변동에 과민 반응. 실제 사람 녹음에서는 더 안정적일 것으로 기대되나, 이 측정 데이터로는 증명 불가. → Q D1 대비: **"TTS 합성 음성 기준이며, 실제 녹음 기준 F1은 추후 마일스톤"** 로 솔직하게.
3. **쟁점 구조화의 확률적 변동** — 동일 입력·동일 모델로 2회 run 시 positions·consensus·decision 결과가 조금씩 달라짐. gemma4:e2b의 temperature 특성. **재현성 문제는 향후 고정 seed + JSON 스키마 강제(pydantic)로 개선 예정**으로 포지셔닝.
