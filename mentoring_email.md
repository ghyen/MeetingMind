## 1. 발화 처리 지연 문제 — 파이프라인 직렬 병목

### 현재 구현

`pipeline.py:171-243`의 `on_utterance()`가 발화 하나에 대해 다음 단계를 순서대로 실행합니다.

```
① DB 저장 (pipeline.py:181)
② STT 교정 (pipeline.py:185, 5개 축적 시 LLM 호출)
③ 토픽 감지 (pipeline.py:190, 3단계 필터 — 3차에서만 LLM 호출)
④ 트리거 감지 (pipeline.py:208, 키워드 기반 — LLM 없음)
⑤ 쟁점 구조화 (pipeline.py:210, 10개 축적 시 LLM 호출)
⑥ 자료 수집 (pipeline.py:212, 매 발화마다 엔티티 추출 LLM → 엔티티별 웹검색)
```

Ollama는 내부적으로 요청을 직렬 처리하기 때문에 `pipeline.py:207`의 분기에서 Ollama일 때는 ⑤, ⑥을 `asyncio.gather`로 묶지 않고 순차 실행하고 있습니다(OpenRouter일 때만 병렬). 최악의 경우 발화 1개당 **LLM이 3회 연속 호출**(토픽 3차 판단 + 쟁점 구조화 + 엔티티 추출)되고, 그 사이 다음 발화 처리가 블로킹됩니다.

`pipeline.py:177`에 이미 `_StepTimer`가 붙어 있어서 단계별 소요 시간이 로그로 찍히고는 있습니다. 실측해보면 발화당 1~4초 정도로 변동이 크고, 토픽 감지 3차 판단과 엔티티 추출이 들어간 발화에서 피크가 발생합니다.

### 고민 중인 방향

**(a) 단계별 우선순위 분리 + 비동기 워커 큐**
- "즉시 UI에 반영되어야 하는 것"(DB 저장, 토픽 감지, 트리거 감지)만 동기 경로에 두고
- "조금 늦어도 되는 것"(쟁점 구조화, 자료 수집, STT 교정)은 별도 asyncio Task 큐로 분리
- 문제: 로컬 LLM이 소화 속도의 상한이라 큐가 밀리면 결국 더 늦어질 수 있습니다. 워커 큐가 밀렸을 때의 백프레셔 정책을 어떻게 설계하시는지가 궁금합니다.

**(b) LLM 호출 횟수 자체 축소 — 프롬프트 통합**
- 토픽 감지 3차 판단과 엔티티 추출은 둘 다 "발화 텍스트를 읽고 구조화된 JSON을 반환"하는 공통점이 있습니다. 하나의 프롬프트로 `{ topic_changed, new_topic_title?, entities[] }`를 한 번에 받으면 LLM 호출을 2회 → 1회로 줄일 수 있을 것 같습니다.
- 우려: 프롬프트가 복합 태스크가 되면 각 서브태스크의 정확도가 떨어질 가능성이 있습니다. 작은 로컬 모델(`gemma4:e2b`)일수록 이 경향이 크다고 알고 있어서 망설여집니다.

**(c) 오디오 스트리밍과 추론의 오버랩** (`TODO.md` #12)
- 현재는 "발화 오디오 축적 완료 → STT → LLM 추론"이 순차적이고, 추론이 도는 동안 다음 발화의 오디오는 수신만 되고 처리는 막혀 있습니다.
- 추론 중에도 다음 발화를 미리 버퍼링·STT까지 선수행해두는 이중 버퍼 구조로 바꾸면, 지연 누적은 완화되지만 근본적인 LLM 처리 속도는 그대로입니다.

### 여쭙고 싶은 점

세 가지 방향 중 실제로 체감 효과가 큰 순서, 그리고 로컬 LLM 환경에서 비슷한 병목을 다뤄보신 경험이 있다면 어떤 방향이 유효했는지 조언 부탁드립니다.

## 2. 웹 검색 과다 호출 문제

### 현재 구현

`pipeline.py:322-342`의 `_search_references()`가 매 발화마다 다음을 수행합니다.

1. `search/__init__.py:45-68`의 `EntityExtractor.extract()` — 발화 텍스트를 LLM에 넘겨 검색 가능한 엔티티 리스트를 뽑습니다. **매 발화마다 LLM 1회**.
2. 각 엔티티에 대해 `ReferenceCollector.search()` 호출 — ChromaDB 사내 검색 + Tavily/DuckDuckGo 웹 검색을 병렬로 돌립니다.

문제는 두 가지입니다.

- **캐싱이 없습니다.** "PG사"가 회의 중 10번 언급되면 LLM 엔티티 추출도 10번, 웹 검색도 10번 새로 돕니다. 같은 결과가 반복적으로 나와 자원도 낭비되고, relevance_score 기준 정렬도 흔들립니다.
- **적합성 판단이 없습니다.** 프롬프트가 "검색 가능한 엔티티"를 추출하도록 설계되어 있어서, 실제 회의에서는 거의 모든 발화에 엔티티가 잡힙니다. "이 수치 근거가 뭐였지?"는 검색이 유용하지만, "Figma에 업데이트해둘게"는 엔티티(Figma)가 추출되더라도 검색이 불필요한데 현재 구조로는 구분할 수 없습니다.

### 고민 중인 방향

**(a) 엔티티 단위 캐싱**
- 정규화된 엔티티 텍스트를 키로 N분간 TTL 캐시
- 구현 부담이 가장 낮음. 다만 "같은 키워드라도 맥락이 달라지면 다른 결과가 필요한 경우"를 놓칠 가능성이 있습니다.

**(b) 발화 의도 분류 — 검색 전 단계**
- 검색 이전에 "이 발화가 정보 요청/근거 탐색 맥락인가, 단순 언급인가"를 판별하는 단계 추가
- 규칙 기반으로 처리 가능한지(의문문 여부, "확인해봐야"·"근거가", 수치 인용 패턴 같은 표지) 아니면 결국 LLM 판단이 필요한 영역인지 판단이 어렵습니다.
- LLM 판단을 추가하면 1번 문제(파이프라인 지연)와 바로 충돌합니다.

**(c) 에이전틱 방식 — LLM이 스스로 tool call로 검색 실행**
- 가장 이상적이지만, `gemma4:e2b` 같은 작은 로컬 모델로 이런 라우팅이 현실적 속도·정확도로 가능한지 확신이 없습니다.

### 여쭙고 싶은 점

- 캐싱 + 규칙 기반 의도 필터 조합이 실무적으로 충분한 수준일지, 아니면 결국 LLM 라우터가 필요한 영역인지
- 로컬에서 LLM 라우팅을 돌린다면 작은 모델로 가능한 실용적 하한선이 어느 정도인지 경험적 감각을 여쭙고 싶습니다.

## 3. 음성 감지(VAD) 방식 선택

### 현재 구현

`stt/whisper_stt.py:116-167`의 `feed_chunk()`에 **RMS 기반 에너지 VAD**가 구현되어 있습니다.

- 초기 2초(4개 청크) 동안 배경 소음 RMS를 측정해 `_noise_floor` 계산(`whisper_stt.py:126-135`)
- `_vad_threshold = max(noise_floor × 3.0, 0.005)`로 적응형 임계값 설정
- 발화 종료 판정: 침묵 `800ms` 이상 (`_silence_threshold_ms`, `whisper_stt.py:49`)

단순하고 빠르지만, 키보드 타이핑·에어컨·종이 넘기는 소리·짧은 박수 등 비음성 임펄스를 발화로 오인하는 경우가 있습니다.

한편 `stt/__init__.py:80-127`에 **Silero VAD 기반 DL VAD**를 래핑한 `VADFilter` 클래스를 따로 구현해두었습니다(`models/silero_vad.onnx` 사용, `config.py:10` 의 `vad_threshold = 0.5`). 다만 `WhisperSTT`가 이걸 호출하지 않고 자체 RMS VAD만 쓰고 있어서 실질적으로 데드 코드입니다.

### 여쭙고 싶은 점

- **에너지 기반이 충분한 환경과 DL 모델이 필요한 환경의 경계**는 어디쯤이라고 보시는지 궁금합니다. 1인 헤드셋 환경은 RMS로 충분할 것 같지만, 회의실·다인·배경 소음 환경에서 Silero 같은 모델이 실효적 차이를 만드는 시점이 있을 것 같습니다.
- **실시간 환경에서 Silero VAD의 추가 연산 비용**이 체감될 정도인지도 궁금합니다. 0.5초 청크마다 한 번씩 DL 추론이 추가되는 부담이 1번의 파이프라인 지연과 누적되는 상황이 우려됩니다.

## 4. 음성 인식 오류 후처리

### 현재 구현

이미 LLM 기반 교정 모듈이 있습니다.

- `analysis/correction.py:27-97` — `STTCorrector` 클래스
- `_BATCH_SIZE = 5` (`correction.py:16`) — 발화 5개가 쌓이면 배치로 LLM 호출
- 프롬프트에 회사/회의 정보(`MeetingContext`)를 컨텍스트로 주입해 도메인 특화 교정 시도 (`correction.py:64-81`)
- 교정 결과는 `pipeline.py:245-261`의 `_correct_stt()`에서 DB 업데이트 + `correction` 이벤트로 UI 브로드캐스트

### 문제점

**(a) 덮어쓰기 UX**
- 현재 흐름은 "원본 텍스트가 UI에 먼저 뜸 → 몇 초 뒤 교정 결과로 덮어쓰기"입니다.
- 사용자 입장에서는 "매줄" → "매출" 같은 변화가 시야에서 일어나서 혼란을 줄 수 있습니다. 반대로 교정을 기다리고 띄우면 실시간성이 깨집니다.

**(b) 단일 모델**
- `correction.py:84`에서 `settings.llm_model_fast`를 쓰지만, `config.py:35-36`에서 `llm_model_fast`와 `llm_model_deep`가 둘 다 `gemma4:e2b`로 같습니다. 사실상 `fast`/`deep` 구분이 이름만 있습니다.

**(c) 지연 추가**
- 5개 배치라 발화 5개마다 LLM 1회 호출이 추가로 누적됩니다. 1번 문제와 동일한 지연 원인이 됩니다.

### 고민 중인 대안

**(a) LLM 없이 경량 교정 — 도메인 용어 사전 기반 매칭**
- 회의 시작 시 도메인(금융/IT/의료/법률 등) 프리셋 사전 로드 → n-gram 유사도·편집 거리 기반 치환
- LLM 호출 0회. 다만 문맥 의존적 교정("매출" vs "매줄")을 규칙만으로 잡을 수 있을지 확신이 없습니다.

**(b) Whisper `initial_prompt` 활용**
- 현재 `whisper_stt.py:83-88`의 `mlx_whisper.transcribe()` 호출에 `initial_prompt`가 비어 있습니다. 도메인 용어를 프롬프트로 주입하면 STT 단계에서 오류 자체가 줄어든다고 들었는데, 실제 한국어 성능 개선이 얼마나 되는지 경험이 있으신지 궁금합니다.

**(c) LLM 교정 유지하되 UX 개선**
- 교정 대기 중임을 명시적으로 표시("교정 중…")하거나, 교정 전/후를 diff로 보여주는 식

### 여쭙고 싶은 점

이 세 방향 중 어느 쪽이 실용적일지, 그리고 한국어 STT에서 `initial_prompt`가 실제로 의미 있는 개선을 만드는지 조언을 구하고 싶습니다.

## 5. 화자 식별 정확도

### 현재 구현

`stt/speaker.py:27-74`에서 `sherpa-onnx`의 `SpeakerEmbeddingExtractor`(3dspeaker ERes2Net)로 발화 단위 임베딩을 추출하고, `SpeakerEmbeddingManager.search()`로 기존 화자 중 가장 유사한 화자를 찾습니다.

- 임계값: `config.py:18`의 `speaker_similarity_threshold = 0.5` — **고정값**
- 매칭 실패 시 새 화자로 등록 (`Speaker 1`, `Speaker 2`, ...)
- 임베딩은 Pipeline 인스턴스 내 인메모리로만 유지 — 회의 간 재사용 안 됨

### 문제점

실제 회의에서 관찰된 두 가지 오류 패턴:

- **Over-merge**: 목소리가 비슷한 두 사람이 하나의 `Speaker`로 묶임
- **Over-split**: 같은 사람이 감정 변화·볼륨 변화로 서로 다른 `Speaker`로 분리됨

두 오류 모두 고정 임계값에서 오는 구조적 한계로 보입니다. 또 **짧은 발화(1초 미만, "네"/"맞아요")** 는 임베딩 자체가 불안정해서 매칭이 흔들립니다.

### 고민 중인 방향

**(a) 임계값 적응적 조정**
- 회의 첫 5분의 발화들로 pairwise 유사도 분포를 구성하고, Otsu's method나 GMM으로 자연 분할점을 찾아 임계값으로 사용
- 참여자 수(2명 vs 5명)에 따라 분포 형태가 달라질 텐데, 단일 방법으로 커버 가능한지 모르겠습니다.

**(b) 짧은 발화 보완**
- 임베딩 길이가 일정 기준 미만이면 "직전 화자 상속" 규칙을 적용
- 다만 끼어들기 상황("네" ← 다른 사람이 말함)에서는 틀린 답이 됩니다.

**(c) 회의 간 임베딩 영속화**
- 같은 팀원들이 반복 회의하는 경우, 화자 임베딩을 DB에 저장해서 재사용하면 누적 정확도가 올라갈 것 같습니다. 다만 회의마다 참여자 구성이 바뀌면 오히려 잘못된 매칭이 될 위험도 있습니다.

### 여쭙고 싶은 점

- 적응적 임계값 조정이 실무에서 체감 효과가 있는 개선인지, 아니면 임계값 튜닝보다 더 본질적인 접근(예: diarization 모델 자체 교체)이 필요한 영역인지
- 짧은 발화 문제에 대한 실용적인 보완 전략

## 6. 동시 발화 처리

### 현재 구현

실시간 경로는 `whisper_stt.py:179-220`의 `_process_utterance()`가 침묵 기반으로 축적한 **전체 오디오 한 덩어리에 대해 화자 식별을 1회만** 수행합니다(`whisper_stt.py:203`).

```python
if settings.diarization_enabled and len(audio) > 1600:
    speaker = self._speaker_id.identify(audio)  # 전체 오디오 → 1명
```

즉, A와 B가 겹쳐서 말한 구간이 있어도 임베딩이 혼합되어 한 명으로 분류됩니다.

한편 파일 업로드 경로(`whisper_stt.py:77-112`)는 whisper 세그먼트 단위로 오디오를 잘라서 세그먼트마다 화자 식별을 따로 수행하기 때문에, 세그먼트 간 화자 전환은 감지됩니다. `TODO.md` #20에 정리된 내용입니다.

### 고민 중인 방향

**(a) 실시간 경로를 파일 경로 구조로 리팩토링**
- `_process_utterance()`에서 mlx-whisper의 세그먼트 경계를 활용해 화자 식별을 세그먼트별로 수행
- 세그먼트 간 전환은 잡히지만, **동일 세그먼트 내 동시 발화**는 여전히 혼합 임베딩 문제로 잡히지 않습니다.

**(b) 전용 speaker diarization 모델 도입** (pyannote.audio 등)
- 정확도는 높지만 Apple Silicon 로컬에서 실시간성 유지가 가능한지 불확실, 추가 연산 비용도 큽니다.

**(c) 실시간은 근사치, 회의 종료 후 일괄 재처리 (offline diarization)**
- 실시간 표시는 부정확함을 감수하고, 종료 후 전체 녹음본으로 화자를 재할당해 최종 회의록을 정확하게 만드는 전략
- 실제 서비스에서 "실시간 부정확 + 최종본 정확"이 자연스러운 UX인지 감이 잡히지 않습니다.

### 여쭙고 싶은 점

실시간 정확도가 꼭 필요한 성격의 서비스인지, 아니면 "실시간 근사 + 오프라인 정확본" 전략이 더 현실적인 선택인지 의견을 여쭙고 싶습니다.

## 7. 쟁점 분석 배치 크기

### 현재 구현

`analysis/issues.py:29-50`의 `IssueStructurer.update()`:

```
새 발화 입력
  │
  ├── 현재 토픽에 IssueGraph 없음 → _create_initial() (LLM 호출, Pro 모델)
  │
  ├── pending 발화 < batch_size → 기존 구조 반환 (LLM 호출 없음)
  │
  └── pending 발화 ≥ batch_size → _apply_delta() (LLM 호출, 점진적 업데이트)
```

- 배치 크기: `config.py:25`의 `issue_batch_size = 10`
- 초기에는 5였는데, 호출 빈도가 너무 높아 10으로 늘렸습니다 (`TODO.md` #21)

### 고민 중인 방향

**(a) 배치 크기 동적 조정**
- 초반 5분: 작게(5) — 쟁점 구조를 빨리 형성
- 이후: 크게(15) — 호출 빈도 감소
- 토픽 전환 직후: 즉시 플러시 — 새 토픽의 초기 쟁점을 빨리 형성

**(b) 토픽 전환 이벤트를 강제 플러시 트리거로 사용**
- 현재는 `_pending[topic.id]`에 쌓인 발화가 토픽 전환 시 처리되지 않고 다음 토픽의 pending과 섞일 가능성이 있습니다. 전환 시점에 강제 플러시가 필요해 보입니다.

### 여쭙고 싶은 점

- 배치 크기 동적 조정이 로컬 LLM 환경에서 의미 있는 개선인지, 아니면 고정값을 잘 정하는 것으로 충분한지
- 로컬 LLM 환경에서는 "호출 빈도 감소(큰 배치)"와 "반영 속도(작은 배치)" 중 사용자 경험에 더 유리한 쪽이 어디인지

### 8. `llm_model_fast`와 `llm_model_deep`의 실질적 분리

`config.py:35-36`에서 두 설정이 모두 `gemma4:e2b`로 동일합니다. 원 설계 의도는 "교정/엔티티 추출은 빠른 모델, 쟁점 구조화/토픽 판단은 더 큰 모델"이었을 텐데, 지금은 이름만 남고 실질적 분리가 없습니다. 로컬 Ollama 환경에서 두 모델을 실제로 분리하면 모델 스와핑 비용(로드 언로드)이 매번 발생해서 오히려 역효과일 것 같은데, 멘토님이라면 이 이원화를 시도하실지 궁금합니다.

### 9. VAD 데드 코드 정리

`stt/__init__.py:80-127`의 `VADFilter`(Silero 기반)는 어디에서도 호출되지 않습니다. 3번 질문과 연결되는데, 이걸 실제 파이프라인에 연결할지 결정하기 전까지는 삭제 또는 주석 정리를 해두는 게 맞을지, 아니면 옵션 스위치로 남겨두는 게 맞을지 의견을 듣고 싶습니다.

### 10. 관측·프로파일링 — 이미 있는 것 활용

`pipeline.py:177`의 `_StepTimer`가 이미 단계별 latency를 찍고 있습니다. 1번에서 "감으로 결정하지 말고 데이터 기반으로 결정해야 한다"는 고민이 컸는데, 사실 로그에는 이미 데이터가 있는 상태입니다. 그런데 로그가 흘러가기만 하고 집계·시각화 기반이 없어서 어느 단계가 누적적으로 병목인지 잘 안 보입니다. 

가장 가볍게 시작할 수 있는 집계 방법으로는 단계별 latency를 SQLite나 JSONL에 append하고 회의 종료 시 평균/p95/p99만 찍어보는 수준을 생각하고 있는데, 이 정도로 의사결정에 충분한지, 아니면 처음부터 더 구조적인 관측 체계가 필요한지도 여쭙고 싶습니다.
