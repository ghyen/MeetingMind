안녕하세요, 멘토님.

MeetingMind 프로젝트를 진행하면서 몇 가지 기술적 판단이 어려운 부분이 있어 질문드립니다.

---

1. 발화 처리 지연 문제

현재 맥북에서 로컬 LLM(Ollama)을 사용하고 있는데, 발화 하나가 들어오면 토픽 감지, 쟁점 분석, 엔티티 추출 등 여러 단계를 순서대로 거쳐야 해서, 앞 발화의 처리가 끝나야 다음 발화 처리가 시작됩니다. 실시간 회의 중에 이 지연이 계속 누적되는 게 문제입니다.

로컬 LLM 환경의 제약 안에서 이 지연을 줄일 수 있는 현실적인 방법이 있을까요? 예를 들어 처리 단계를 줄이거나, 우선순위를 나누거나, 비동기 처리를 도입하는 방식 등을 고민하고 있습니다.

2. 웹 검색 과다 호출 문제

매 발화에서 고유명사나 수치 같은 키워드가 추출되면 무조건 웹 검색을 실행하고 있습니다. 그런데 실제 회의에서는 거의 모든 발화에 이런 키워드가 포함되어 있어서, 대부분의 발화마다 검색이 발생합니다. 같은 키워드가 반복 언급되어도 매번 새로 검색하고요.

"이 발화는 정말 검색이 필요한가"를 판별하는 기준을 어떻게 설계하면 좋을까요? 단순히 키워드 유무가 아니라, 발화의 의도(정보를 요청하는 건지, 단순 언급인지)나 이전 검색과의 중복 여부 같은 기준이 필요할 것 같은데, 어떤 접근이 적합할지 조언을 구하고 싶습니다.

또한, LLM이 스스로 검색 필요 여부를 판단하는 에이전트 방식도 고려하고 있는데, 로컬 LLM 환경에서 이런 방식이 현실적인지도 궁금합니다.

3. 음성 감지(VAD) 방식 선택

현재는 소리의 크기(에너지)만으로 음성 여부를 판단하고 있습니다. 간단하긴 하지만, 키보드 소리나 에어컨 같은 환경 소음을 음성으로 오인하는 경우가 있습니다. 별도로 딥러닝 기반 음성 감지 모델을 연동하는 코드를 작성해둔 상태인데, 실제 파이프라인에는 아직 연결하지 않았습니다.

에너지 기반 방식으로 충분한 상황과 전용 모델이 필요한 상황의 경계가 어디쯤인지, 그리고 실시간 환경에서 전용 모델의 추가 연산 비용이 체감될 정도인지 궁금합니다.

4. 음성 인식 오류 후처리

음성 인식 결과에 오류가 꽤 있습니다. 예를 들어 "매출"이 "매줄"로, "결재"가 "결제"로 인식되는 식입니다. 이 오류가 이후 분석 전체에 영향을 줍니다.

LLM으로 교정하면 정확도는 올라가지만 처리 시간이 더 늘어나고, 안 하면 분석 품질이 낮습니다. LLM 없이 경량으로 교정하는 방법(용어 사전 기반 매칭 등)이 현실적인지, 아니면 LLM 교정을 넣되 지연을 최소화하는 전략이 있는지 조언 부탁드립니다.

5. 화자 식별 정확도

화자를 구분할 때 음성 특징의 유사도를 수치로 비교하는데, 그 기준값을 고정해두고 있습니다. 문제는 목소리가 비슷한 사람들이 같은 화자로 묶이거나, 반대로 같은 사람이 톤 변화로 다른 화자로 분리되는 경우가 있다는 점입니다.

회의 환경에 따라 이 기준값을 적응적으로 조정하는 방법이 있을까요? 그리고 아주 짧은 발화는 화자 식별이 실패하는데, 이를 보완하는 전략도 궁금합니다.

6. 동시 발화 처리

현재 화자 식별은 sherpa-onnx의 3dspeaker 모델로 발화 단위 임베딩을 추출한 뒤 유사도를 비교하는 방식입니다. 그런데 두 명 이상이 동시에 말하면, 임베딩이 섞여서 한 명의 화자로만 인식됩니다. 실제 회의에서는 끼어들기나 동시 발화가 꽤 빈번한데, 실시간 환경에서 이를 처리하는 현실적인 방법이 있는지, 아니면 회의 종료 후 일괄 재처리하는 게 더 적합한지 궁금합니다.

7. 쟁점 분석 배치 크기

쟁점 분석을 할 때 발화 5개가 쌓일 때마다 한 번에 처리하고 있습니다. 배치를 줄이면 실시간성은 좋아지지만 LLM 호출이 잦아지고, 늘리면 호출은 줄지만 반영이 느려집니다.

배치 크기를 상황에 따라 동적으로 조정하는 게 유효한 전략인지, 그리고 로컬 LLM 환경에서는 어느 쪽이 사용자 경험에 더 나을지 의견을 여쭙고 싶습니다.
