### 제약 조건

### 질문

## Q2. 발화별 웹 검색 과다 호출 — 검색 적합성 판단 기준 설계

### 현재 상황

- 매 발화마다 `EntityExtractor`(LLM)가 엔티티를 추출하고, 엔티티가 1개라도 있으면 **무조건 웹+사내 검색 실행**
- 검색 여부를 판단하는 로직은 LLM 프롬프트 하나뿐이며, 별도의 필터나 스코어링이 없음

```
발화 → EntityExtractor(LLM) → 엔티티 있음? → 무조건 Tavily/DuckDuckGo + ChromaDB 검색
                               엔티티 없음? → 스킵
```

- EntityExtractor 프롬프트의 추출 대상이 넓음: "문서, 데이터, 수치, 인물, 조직"
- 예시: `"Figma에 업데이트해둘 테니까"` → Figma를 엔티티로 추출 → 웹 검색 실행

### 문제점

1. **과다 검색** — 실제 회의에서 고유명사·수치 언급이 빈번하여, 대부분의 발화에서 엔티티가 추출되고 웹 검색이 발생
2. **중복 검색** — 같은 엔티티("Figma", "PG사")가 여러 발화에 걸쳐 반복 언급되어도 매번 새로 검색
3. **LLM 호출 추가 부담** — EntityExtractor 자체가 LLM 호출이므로, Q1의 직렬 병목 문제를 악화시킴 (Ollama 환경에서 파이프라인당 LLM 호출 4회: 토픽감지 + 쟁점구조화 + 엔티티추출 + ...)
4. **검색 품질** — "회의 참석자에게 실제로 유용한 검색"과 "엔티티가 언급되었으니 일단 검색"의 차이가 큼

### 고려 중인 접근

- **Agentic 방식**: LLM에 검색 도구를 tool로 제공하고, LLM이 "이 발화는 검색이 필요하다"고 판단할 때만 도구를 호출하도록 하는 방식
  - 단, 현재 Ollama(로컬) 환경에서는 병렬 추론 불가 → window/vLLM 전환도 검토 중
  - Ollama에서는 tool use(function calling) 지원이 제한적일 수 있음

### 제약 조건

- MacBook + Ollama 기반 (Q1과 동일, vLLM 전환 검토 중)
- 검색 API 호출 비용 (Tavily 유료, DuckDuckGo 레이트 리밋)
- Q1의 직렬 병목과 연관 — 불필요한 LLM 호출·검색을 줄이면 파이프라인 전체 지연도 감소

### 질문

1. 회의 발화에서 "웹 검색이 실제로 유용한 발화"를 판별하는 기준을 어떻게 설계할 수 있는가?
   - 엔티티 존재 여부만으로는 부족한데, 어떤 추가 기준이 필요한가?
   - (예: 발화의 의도 — 정보 요청 vs 단순 언급, 기존 검색 결과와의 중복 여부, 토픽 전환 시점 등)
2. Agentic 방식(LLM이 tool use로 검색 호출을 자율 판단)이 이 문제에 적합한 접근인가?
   - 로컬 LLM 환경(Ollama/vLLM)에서 tool use 패턴을 적용할 때의 현실적 제약은?

## Q3. RMS 기반 VAD vs 전용 VAD 모델 — 실시간 음성 감지 정확도

### 현재 상황

- `WhisperSTT.feed_chunk()`에서 **RMS(에너지) 기반 VAD**를 자체 구현하여 사용 중
  - 2초간 배경 소음 측정 → 노이즈 플로어 × 3.0 = 음성 판별 임계값
  - 임계값 초과 = 음성, 이하 = 침묵 → 1.2초 연속 침묵이면 발화 종료
- 별도로 `stt/__init__.py`에 **Silero VAD** (`VADFilter` 클래스)가 이미 구현되어 있으나, `WhisperSTT`에서 사용하지 않음

```python
# whisper_stt.py — 실제 사용 중 (RMS)
is_speech = rms > self._vad_threshold  # 단순 에너지 비교

# stt/__init__.py — 구현만 되어 있고 미사용 (Silero VAD)
class VADFilter:
    def is_speech(self, audio_chunk: bytes) -> bool:
        # sherpa_onnx.VadModel 기반 뉴럴 VAD
```

### 문제점

1. RMS VAD는 환경 소음(음악, 키보드, 에어컨)을 음성으로 오판할 수 있음
2. 노이즈 플로어 × 3.0 배수의 근거가 불명확 — 환경에 따라 최적값이 다름
3. Silero VAD가 이미 구현되어 있는데 왜 통합하지 않았는지 (성능? 지연? 의도적 선택?)

### 질문

1. RMS VAD로 충분한 상황과 전용 VAD 모델이 필요한 상황의 경계는?
2. 실시간 환경에서 Silero VAD의 추가 연산 비용이 RMS 대비 얼마나 되는가? 지연에 영향이 있는가?

## Q5. 화자 식별 임계값(cosine similarity 0.5) 튜닝 전략

### 현재 상황

- `SpeakerIdentifier`가 3dspeaker(sherpa-onnx) 임베딩을 추출하고, 등록된 화자와 cosine similarity 비교
- 유사도 ≥ 0.5 → 기존 화자, 미만 → 새 화자 등록
- 임계값 0.5가 `config.py`에 하드코딩

```python
# speaker.py:65
name = self._manager.search(embedding, settings.speaker_similarity_threshold)  # 0.5
```

### 문제점

1. **과소 분리**: 임계값이 낮으면 다른 화자를 같은 사람으로 오인 (목소리가 비슷한 경우)
2. **과다 분리**: 임계값이 높으면 같은 화자가 감정·톤 변화로 다른 사람으로 등록됨
3. 회의 참여자 수, 성별 구성, 마이크 거리에 따라 최적 임계값이 달라짐
4. 짧은 발화(0.1초 미만)는 임베딩 추출 불가 → `_fallback_label()`로 마지막 화자 재사용 → 화자 붕괴 가능

### 질문

1. cosine similarity 임계값을 회의 환경에 맞게 적응적으로 조정하는 방법은?
   - (예: 회의 시작 시 참여자별 음성 등록, 화자 수 사전 입력, 클러스터링 기반 동적 조정 등)
2. 짧은 발화의 화자 식별 실패를 보완하는 전략은?
