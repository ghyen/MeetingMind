### Q1. "원칙 4개가 전부 'LLM 덜 쓰기' 아닌가요? 왜 4개로 나눴어요?"

#### 🎯 30초 핵심
> "뿌리는 같지만 **작동하는 축이 다릅니다**. 원칙 1은 '순서', 원칙 2는 '빈도', 원칙 3은 '대체', 원칙 4는 '병렬성'. 그리고 말씀하신 대로 원칙 1과 3은 겹치는 면이 있어서, 발표에선 계층형 판정 하나로 묶고 **3개로 정리**하는 것이 더 정직하다고 생각합니다."

#### 📜 확장 스크립트 (1분)
> "좋은 질문입니다. 솔직히 말씀드리면, 원칙 1(Cheap-First)과 원칙 3(도메인 구조 활용)은 **같은 뿌리**에서 나왔습니다. '비싼 연산 전에 값싼 판단' 이라는 공통점이 있습니다.
>
> 다만 각 원칙이 **제어하는 축이 다르다**고 보고 나눴습니다.
> - 원칙 1은 **판정 시퀀스** — '어떤 순서로 검사할 것인가'
> - 원칙 2는 **판정 빈도** — '얼마만큼 정보가 쌓이면 판정할 것인가'
> - 원칙 3은 **판정 주체** — 'LLM인가 코드인가'
> - 원칙 4는 **판정 결과 전달 방식** — 'UI에 언제 반영할 것인가'
>
> 이 중 1+3은 공통 뿌리가 있어서, 발표용으로는 **'계층형 판정: 키워드→상태→LLM'** 하나로 통합하고, 2와 4를 독립 원칙으로 두는 3개 구조가 더 깔끔합니다. 질문해 주신 덕분에 이 부분을 더 간결하게 정리해야겠다고 느꼈습니다."

### Q2. "540회 계산이 순차 가정인데, 병렬로 쏘면 되지 않나요?"

#### 🎯 30초 핵심
> "맞는 지적이고, 실제로 OpenRouter 같은 외부 API에선 병렬화로 해결합니다. 하지만 우리는 **로컬 단일 모델 인스턴스**가 제약이라, 병렬로 쏴도 Ollama 내부에서 **직렬로 큐잉**됩니다. 코드에도 이 조건 분기가 들어가 있습니다."

#### 📜 확장 스크립트 (1분 30초)
> "실은 이 질문이 제가 가장 대비한 질문 중 하나입니다. 말씀하신 대로 540회를 순차로 처리한다고 가정하면 naive한 계산이고, **병렬화**라는 빤한 반론이 나옵니다.
>
> 저희의 답은 이렇습니다. **병렬화 자체는 이미 코드에 있습니다.** `pipeline.py`의 `on_utterance`를 보시면, 외부 API(OpenRouter) 사용 시엔 쟁점 구조화와 자료 수집을 `asyncio.gather`로 동시 실행합니다.
>
> 하지만 **Ollama 로컬 인스턴스는 본질적으로 직렬**입니다. GPU/CPU 하나를 공유하는 단일 프로세스라, `asyncio.gather`로 요청을 쏴도 Ollama 내부 큐에서 **순서대로** 처리됩니다. 오히려 병렬로 쏘면 컨텍스트 스위칭 오버헤드로 더 느려집니다.
>
> 그래서 저희 파이프라인은 **provider 분기**가 있습니다. `_active_provider`가 `ollama`나 `bonsai`면 순차 실행, OpenRouter 같은 외부 API면 병렬 실행. 이 분기 자체가 '로컬 LLM은 병렬로 못 푼다'는 전제를 코드로 명시한 겁니다.
>
> 즉, 540회의 본질은 '병렬화로 풀리는 문제'가 아니라 **'단일 모델 인스턴스의 총 처리량 한계'** 입니다. 이걸 뚫으려면 모델 인스턴스를 늘려야 하는데, 로컬 머신 한 대라는 요구사항과 충돌합니다. 그래서 **호출 횟수 자체를 줄이는 방향**으로 문제를 재정의했습니다."

#### 🧱 근거 코드
- Provider 분기: `pipeline.py:232-263`
- Ollama 순차 실행 주석: `pipeline.py:230-231` ("Ollama는 내부적으로 요청을 직렬 처리하므로 병렬 호출해도 이득이 없음")

#### ⚔️ 후속 공격 대비
- **"vLLM 같은 동시성 있는 엔진 쓰면 되잖아요?"** → "vLLM은 GPU가 충분할 때 KV 캐시 공유로 이득이 큽니다. Apple Silicon 로컬 환경에선 MLX 기반이 더 유리하고, 현재 bonsai가 `gemma4-e2b-mlx`를 쓰고 있습니다. 다만 **vLLM 수준의 continuous batching은 로컬 단일 머신에선 한계가 있고**, 결국 호출 횟수 자체가 본질 병목이라는 결론이 달라지지 않습니다."
- **"Ollama 인스턴스를 여러 개 띄우면?"** → "메모리가 모델 크기 × N으로 늘어나, 로컬 머신에선 현실적이지 않습니다. 또한 같은 GPU를 공유하므로 실질적 동시성도 제한적입니다."

### Q4. "gemma4:e2b로 한국어 쟁점 구조화 품질 나오나요?"

#### 🎯 30초 핵심
> "품질 **정량 측정은 아직**입니다. 대신 설계 레벨에서 **구조적 실패를 방지**하는 3겹 방어를 넣었습니다: JSON 스키마 강제 + few-shot 예시 + 화자별 병합 후처리. 모델이 틀려도 **포맷은 절대 안 깨지도록** 구축했고, 모델 교체는 config 한 줄로 가능합니다."

#### 📜 확장 스크립트 (1분 30초)
> "두 부분으로 나눠서 답변드리겠습니다.
>
> **1. 현재 품질 수준** — 솔직히 말씀드리면 gemma4:e2b는 **작은 모델**이라 한국어 쟁점 추론에서 완벽하지 않습니다. '입장 통합'이나 '합의 판단' 같은 추론적 작업에서 가끔 엉뚱한 결과가 나옵니다. 다만 이건 gpt-4 같은 클라우드 모델 대비 품질이고, **로컬 퍼스트 제약** 하에서 받아들인 트레이드오프입니다.
>
> **2. 품질 하락을 완화하는 설계** — 모델 품질이 낮을 수록 구조적 방어가 중요합니다.
> - **JSON 스키마 강제**: 프롬프트에 출력 포맷을 명시하고, `analysis/llm.py`의 `ask_json`이 JSON 파싱 실패 시 폴백 처리합니다.
> - **Few-shot 예시**: `issues.py`의 `_FEW_SHOT_EXAMPLE` 하나로 모델이 어떤 수준의 stance/argument를 뽑아야 하는지 학습합니다.
> - **화자별 강제 병합**: LLM이 같은 화자를 여러 Position으로 뱉어도 `_merge_positions`에서 코드 레벨로 1인 1 Position 보장합니다. `issues.py:120-147`에 있습니다.
>
> **3. 업그레이드 경로** — `config.py`의 `llm_model_fast`, `llm_model_deep`을 바꾸면 즉시 다른 모델로 교체됩니다. 예를 들어 qwen2.5-14b나 Gemma 27b로 업그레이드하면 품질이 즉각 올라갑니다. **인터페이스를 깨지 않고 모델만 바꿀 수 있는 것**이 이 설계의 기여입니다.
>
> 그래서 제 입장은: '지금 품질은 완벽하지 않지만, **품질이 모델 교체로 개선되는 구조**를 갖춘 것이 본질적 작업이었다'입니다."

### Q5. "정확도 측정은요?"

#### 🎯 30초 핵심
> "**이번 프로젝트 범위는 '측정 가능한 인프라 구축'까지로 의도적으로 한정했습니다.** F1, precision, recall은 라벨링 데이터셋이 있어야 의미 있는 숫자가 나오고, 이건 별도 프로젝트 단위입니다. 다음 마일스톤이 **회의 3~5건 수동 라벨링 → F1 → 파라미터 스윕** 순서입니다."

#### 📜 확장 스크립트 (스코프 방어형, 1분)
> "정직하게 답변드리겠습니다. **F1, precision, recall 같은 정량 지표는 이번 프로젝트에서 측정하지 못했습니다.**
>
> 이유는 두 가지입니다.
>
> **1. 라벨링 데이터 부재** — 한국어 회의의 '올바른 쟁점 구조'를 정의하려면 실제 회의 3~5건을 누군가 수동으로 라벨링해야 합니다. 이건 짧아도 1~2주, 길면 한 달짜리 별도 작업입니다.
>
> **2. 의도적 스코프 결정** — 만약 제가 라벨링에 1개월을 쓰고 남은 시간으로 시스템을 만들었다면, 지금 보여드리는 '회의 역학 실시간 분석'이라는 결과물 자체가 없었을 겁니다. 그래서 **우선순위를 '일단 측정 가능한 시스템을 만든다'** 로 잡았습니다.
>
> **3. 지금 준비된 것** — 정량 측정은 못 했지만, 측정의 **기반**은 다 깔려있습니다.
> - `_StepTimer`로 성능 측정 인프라
> - DB에 모든 발화/토픽/쟁점/트리거 영구 저장 — **재라벨링 없이도** 분석 가능

>

### Q6. "Doherty 400ms 출처?"

#### 🎯 30초 핵심
> "Doherty & Thadani, 1982 IBM 연구입니다. 다만 이건 **역사적 앵커**고, 현대 UX 기준은 Nielsen의 100ms/1s 기준이 더 엄밀합니다. 저희가 400ms를 **목표 상한**으로 잡은 실용적 이유는 **STT 자체 지연이 150~300ms**라, 그 뒤 버퍼까지 고려한 보수적 기준이기 때문입니다."

#### 📜 확장 스크립트 (용어 논쟁 회피형, 45초)
> "솔직히 'Doherty threshold'라는 용어는 **완화해서 표현하는 게 맞다**고 생각합니다.
>
> Doherty-Thadani (1982, IBM)는 '시스템 응답 <400ms일 때 사용자 사고 속도가 배가된다'는 연구입니다. 40년 전 메인프레임 시대 기준이라, 현대 웹 UX엔 Nielsen의 100ms(즉각 반응), 1초(무리 없음)가 더 널리 쓰입니다.
>
> 저희가 400ms를 **목표 한계**로 잡은 건 학술적 엄밀성보다 **실무적 이유**입니다:
> - STT(음성 인식) 자체 지연이 150~300ms
> - 네트워크 왕복 + WebSocket 직렬화 ~30ms
> - 남는 여유분 ~50~250ms에서 DB 저장과 트리거 감지 수행
>
> 즉, 400ms는 **'사용자가 말을 끝낸 시점부터 화면에 transcript가 뜨는 시점까지'** 의 총 예산입니다. 용어를 순화한다면 '**첫 피드백 400ms 이하**' 정도로 표현하는 게 정확합니다."

### Q10. "WebSocket 연결 끊기면? 재접속 시 상태 복원은?"

#### 🎯 30초 핵심
> "현재 구현은 **DB 기반 상태 영속화 + 재접속 시 최근 상태 재조회**입니다. 모든 발화/토픽/쟁점/트리거가 DB에 즉시 저장되므로, 재접속 클라이언트는 DB에서 현재 meeting의 전체 상태를 로드해서 화면 복원합니다. WebSocket 자체 재연결은 **클라이언트 측 exponential backoff**으로 처리합니다."

#### 📜 확장 스크립트 (45초)
> "두 레벨로 답변드리겠습니다.
>
> **1. 서버 측 영속화** — 파이프라인의 모든 주요 이벤트는 **즉시 DB에 저장**됩니다. 발화(`save_utterance`), 토픽(`save_topic`), 쟁점(`save_issue`), 트리거(`save_intervention`), 자료(`save_reference`) — 전부 WebSocket push와 **독립적**으로 저장됩니다. 즉, 연결이 끊겨도 **데이터 유실은 없습니다**.
>
> **2. 클라이언트 재접속** — React 클라이언트가 WebSocket 연결 끊김을 감지하면 현재 meeting_id 기준으로 REST API(`/api/meetings/{id}/...`)를 호출해서 상태를 다시 불러옵니다. 그다음 새 WebSocket을 열면 **델타만** 받습니다.
>
> **3. 정직한 한계** — 음성 스트리밍 자체는 재접속 시 **그 사이 발화는 유실**됩니다. 즉, 분석 결과는 복원되지만 끊긴 동안의 실시간 오디오는 다시 녹음됩니다. 이건 로컬 브라우저 오디오 캡처의 본질적 한계입니다.
>
> 완벽한 exactly-once 시스템은 아니고, **'분석된 것까지는 복원 보장'** 수준입니다."

- DB 저장 동기성: `pipeline.py:207-214` (첫 단계가 DB 저장)
- WebSocket disconnect 핸들러: `api/websocket.py:45-47, 178-190`
