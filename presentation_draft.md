# MeetingMind 발표 시나리오 (v4)

> 목적: 기업 채용 과제 발표 (15-20분)
> Tagline: **속도 최적화를 "모델을 빠르게"가 아니라 "호출 구조를 바꾸기"로 재정의하는 엔지니어**
> v4 수정 원칙: 실측 우선 · 원칙 축약(4→3) · 비즈니스 맥락 명시 · 코드-발표 일치

## 0. 오프닝 (2분)

### [1] 회의, 이런 경험 있으시죠? (50초)

회의 끝나고 복도에서, 이런 말 한 번쯤 해보셨을 겁니다.

> **"오늘 결론이 뭐였지?"**

- 30분 회의 중 절반은 이미 했던 얘기가 **다시** 나옴
- 누가 무엇에 찬성·반대였는지 회의가 끝나야 정리됨
- "그래서 정해진 거야, 안 정해진 거야?" 애매한 채로 해산
- 특히 민감한 회의 — 계약 협상·평가·이사회 — 는 **녹음조차 조심스러움**

이런 경험, 다들 겪어보셨을 겁니다. 그리고 지금까지 이걸 **실시간으로** 풀어주는 도구는 없었습니다 — 특히 **로컬에서**.

### [2] 그래서 이걸 만들었습니다 (50초)

**[30초 데모 영상 재생]**

이게 MeetingMind입니다.

민감한 회의를 **클라우드에 올리지 않고**, 12.5분 회의 1건 실측에서 단순 발화당 3회 호출 가정 522회 대비 **LLM 실호출 4회 — 절감률 99.2%** [실측] 로 실시간 쟁점 구조화를 구동합니다.

### [3] 오늘의 초점 (20초)

오늘 1순위로 말씀드리고 싶은 건 **"어떻게"** 입니다.

> 속도 최적화를 **"모델을 빠르게"가 아니라 "호출 구조를 바꾸기"** 로 재정의한 과정을 보여드리겠습니다.

## 1. 문제 공백 — 기존 도구가 못 하는 것

현재 시중 회의 AI는 이렇게 접근합니다:

| 도구 유형 | 제공 | 못 하는 것 |
|---|---|---|
| Zoom AI Companion, 클로바노트 | 사후 전사 + 요약 | 실시간 구조화 |
| 일반 회의록 도구 | 발언 기록 | "지금 논의가 어디로 가는가" |
| 퍼실리테이터(사람) | 구조 관리 | 스케일·비용 |

공백은 여기 있습니다:

> **"지금 이 순간, 누가 무슨 입장이고, 합의에 가까운가 교착인가"**
> 를 실시간으로 알려주는 도구가 없다.

회의 "녹화"는 있지만 회의 "**역학(dynamics)**"을 실시간으로 이해하는 도구는 없습니다.

### 네 가지를 실시간으로 추적
- **토픽**: 지금 무엇에 대해 이야기 중인가
- **입장**: 누가 어떤 주장인가
- **긴장**: 반대·교착 여부
- **결정**: 합의/유보

### 세 가지 제약의 충돌
| 제약 | 요구 | 충돌 상대 |
|---|---|---|
| **로컬 퍼스트** | 민감 데이터 — 외부 API 금지 | 성능 |
| **실시간성** | 첫 피드백 400ms 이내 | 정확도 |
| **다기능** | 토픽/쟁점/트리거/자료 | 자원 |

이 셋이 동시에 충족되어야 하는 것이 MeetingMind의 근본 과제입니다.

## 2. 비즈니스 맥락 — 왜 이 문제가 돈이 되는가

### 타겟
민감 정보를 다루는 회의:
- **법무·M&A**: 계약 협상, 실사
- **인사**: 평가·징계 논의
- **이사회·전략**: 비공개 의사결정

### 경쟁 구도
| | 실시간 | 로컬 | 역학 분석 |
|---|---|---|---|
| Otter.ai, Fireflies | ✅ | ❌ | ❌ |
| 클로바노트 | 부분 | ❌ | ❌ |
| Zoom AI Companion | 부분 | ❌ | ❌ |
| **MeetingMind** | ✅ | ✅ | ✅ |

**로컬 퍼스트는 "비용 절감"이 아니라 진입 요건(compliance requirement)** 입니다. 법무·의료·금융에서 클라우드 STT는 아예 선택지가 아닙니다. 이 구간의 고객은 "좋으면 쓴다"가 아니라 **"로컬이어야만 쓸 수 있다"**.

## 3. 핵심 통찰 — LLM을 "언제 쓰지 않을지"

### 자연스러운 첫 번째 접근

회의 이해에는 LLM이 필요합니다. 그러면 이런 질문이 나옵니다:

> **"어떻게 LLM을 더 빠르게 돌릴까?"**

| 시도 | 방법 | 결과 |
|---|---|---|
| STT 가속 | mlx-whisper (Apple Silicon 최적화) | 속도 개선 확인 |
| 모델 경량화 | `benchmark_cohere_4bit.py`, `benchmark_whisper_4bit.py` | 성능 측정 |
| 추론 엔진 | Ollama + **gemma4:e2b (mlx 최적화)** | 로컬 구동 가능 |

빠른 진전처럼 보였습니다. 그런데 계산을 해봤습니다.

### 계산이 드러낸 현실

```
30분 회의 × 발화 180개 × 3종 LLM 분석 = 호출 540회
```

여기서 당연히 나올 반론:
> **"이거 순차 가정 아닌가요? 병렬로 쏘면 되잖아요."**

답: Ollama 로컬 인스턴스는 1개, GPU/CPU 공유 상태에서 동시 요청은 **큐잉**됩니다. 동시 4개 요청 기준 지연이 선형적으로 증가합니다. 초당 1~2 inference가 한계.

→ **병렬화는 자원이 무한할 때만 유효**합니다. 로컬 단일 모델은 본질적 직렬화 제약.
→ "540회"는 동시성 기술로 풀리는 문제가 아닙니다.

### 질문 전환

**잘못된 질문**: "어떻게 LLM을 더 빠르게 만들까?"
→ 양자화·추론 엔진 교체. 호출 **횟수**가 병목이라 본질적 한계 못 넘음.

**올바른 질문**: **"회의의 어떤 순간에, LLM 없이 판단이 가능한가?"**
→ 도메인 구조 분석 → 대부분의 결정은 결정론적 패턴.

> "언제 검색해야 하는가", "언제 토픽이 바뀌었는가" 같은 결정은
> **언어 이해가 아니라 패턴 감지** 문제입니다.

이 질문 전환이 이후 설계 전체를 바꿨습니다.

## 4. 설계 원칙 3가지

### 원칙 1. 계층형 판정 (키워드 → 상태 → LLM)

> 비싼 판단 전에 값싼 판단을 먼저 통과시킨다.
> 도메인 구조가 이를 가능하게 만든다.

**관찰 — 회의 역학의 많은 부분은 패턴 감지 문제**:
- "합의"는 "동의합니다 / 그렇게 하죠 / 좋습니다" 등 **제한된 표현**으로 수렴
- "교착"은 **같은 단어 반복**이라는 외형적 신호
- "결정 없이 넘어감"은 **토픽 전환 + 결정 필드 비어있음** 상태 결합
- "시간 초과"는 **타임스탬프 비교**

전부 키워드·카운트·타임스탬프로 감지 가능합니다.

**적용**:
- 토픽 감지 3단계 필터 (키워드 → 복합키워드 → LLM) + N발화(=12)마다 강제 LLM 판정으로 누탐 방지
- 6종 개입 트리거 전부 LLM 0회
- 자료 수집: 쟁점 구조가 실제 갱신될 때만 1회 검색 (= 원칙 2의 토큰 배치를 자료수집까지 확장)

**효과**:
- LLM이 "판정 주체"에서 **"예외 처리기"** 로 역할 전환
- 판정 지연: 정규식+카운터 연산만이라 **이론상 <1ms** (실측은 `_StepTimer`로 다음 단계)
- 트리거가 LLM 비용에 묶이지 않으므로 **자주 실행해도 부담 없음**

**의식한 약점**: 키워드는 "생각 못 한 표현"에 취약 → 운영 중 오탐/누탐 케이스를 로그에서 수집해 사전(불용어 ~50개, 확장 구조)에 계속 추가. 제품 운영 과제로 명시.

---

### 원칙 2. 정보량 기반 트리거 (이산 → 연속)

> "몇 번 일어났는가"가 아니라 "얼마나 쌓였는가"로 판단한다.

**동기**: "발화 10개마다 호출"은 **정보량의 편차를 무시**함.
- "네", "맞아요", "그렇죠" × 10개 → 거의 정보 없음
- 긴 논증 2개 → 실질 내용 풍부

**해결**: 발화 수 → **토큰 누적량** 기준 배치
- 500토큰 (tiktoken `cl100k_base` 기준)
- 임계 도달 시 쟁점 구조화 배치 + delta 반영

**효과**:
- 정보 밀도 낮은 구간에서 LLM 자원 절약
- 밀도 높아지면 자연스럽게 분석 빈도 증가 → **적응형 동작**
- 구조가 **튜닝 가능성**을 설계에 내장

500토큰은 경험적 출발점이고, 최적값은 실사용 데이터로 스윕해야 합니다. 이를 **가능하게 만든 구조 자체**가 설계 기여입니다.

---

### 원칙 3. 비동기 디커플링 (사용자 체감 ≠ 시스템 완료)

> 400ms라는 반응성 기준은 "모든 처리 완료"가 아니라 "첫 피드백"의 기준이다.

Doherty-Thadani(1982, IBM)가 출발점이지만, 이후 UX 연구에서 **100ms=즉각, 1초=무리없음(Nielsen)** 으로 수렴해왔습니다. STT 자체 지연이 150-300ms라 **400ms를 첫 피드백의 보수적 목표**로 잡았습니다.

**설계**: 파이프라인을 두 개의 push로 분리
- `transcript` push → STT 완료 즉시 (100~300ms) → **첫 피드백 400ms 이내**
- `analysis` push → `asyncio.create_task`로 LLM 백그라운드 → 완료 시점에 추가 push

**효과**:
- LLM 호출이 필요한 경로도 **체감상 느리지 않음**
- 분석 품질을 타협하지 않으면서 UX 확보 → **"속도 vs 정확도" 이분법 탈출**

---

### 원칙 → 모듈 매핑

| 모듈 | 적용 원칙 | 핵심 설계 선택 |
|---|---|---|
| 토픽 감지 | 1 | 3단계 필터. LLM은 fallback |
| 쟁점 구조화 | 2 | 토큰 누적 배치 + delta 반영 |
| 개입 트리거 | 1 | 6종 전부 키워드/카운트/타임스탬프 |
| 자료 수집 | 2 | 쟁점 구조 갱신 시에만 1회 (토큰 배치 트리거 재사용) |
| 파이프라인 | 3 | transcript/analysis 분리 push |

**자료 수집 포인트**: "언제 검색할지"를 LLM에게 묻지 않고 **쟁점 구조가 실제 갱신된 순간**에만 트리거 — 원칙 2의 토큰 누적 배치를 자료 수집까지 확장. 발화당 키워드 추출·검색을 제거해 호출 빈도를 쟁점 단위로 압축.

## 5. 실측 결과 & 측정 인프라

### `_StepTimer`: 단계별 시간 가시화

모든 발화의 단계별 시간을 로깅:

```
[파이프라인 발화#7] 총 1.23초 |
  DB저장 0.01s (1%) | STT교정 0.00s (0%) |
  토픽감지 0.02s (2%) | 트리거감지 0.01s (1%) |
  쟁점구조화 0.00s (0%) | 자료수집 1.19s (97%)
```

→ **병목이 어디인지 가시화** — 측정 인프라를 첫 기능으로 넣었습니다
→ 경로별 분포·최적화 전후 비교 가능

### 실측 결과 [2026-04-24, meeting_id=61]

> 주의: 본 측정은 자료 수집이 부동의 게이트(`positions≥2 && consensus==null`)로 트리거되고 쟁점구조화·자료수집이 `asyncio.gather`로 병렬 실행되던 시점의 결과입니다. 이후 자료 수집 트리거를 **쟁점 구조 갱신 시 1회**로 단순화하면서 두 단계는 직렬화됐고, 자료 수집 호출 횟수와 평균 처리 시간이 추가로 줄어들 것으로 예상됩니다(다음 실측 마일스톤에서 갱신 예정).


**측정 조건**: edge-tts 합성 12.5분 회의 오디오 (4 voice, 86 스크립트 발화 → STT 후 174 발화, 안건 3개), Ollama `gemma4:e2b-it-q4_K_M`, `MM_TIME_OVER_ALERT_MIN=4.0`

| 쟁점 구조화 LLM 호출 | **2회** (토픽당 초기 구조 1회씩) |

| DB저장 | 0ms | 0% |
| **발화당 총합** | 평균 **1.33s** / 중앙 **0.01s** / 최대 **12.47s** | — |

**핵심 발견**: 토픽감지(1ms)·트리거감지(0ms)가 전체의 **0.1% 미만** — 경로 A/B/C 설계가 실제로 LLM 호출을 걷어냈다는 **직접 증거**. 남은 시간은 실제 LLM이 필요한 곳(자료수집·쟁점구조화)에만 쓰입니다. 중앙값이 0.01s라는 건 **대다수 발화는 사실상 비용 0** — 쟁점구조화는 500토큰 배치로 누적되고 자료수집은 당시 부동의 게이트로 차단되기 때문. 측정 시점에는 `asyncio.gather`로 쟁점구조화·자료수집을 병렬 실행하여 직렬 버전 대비 평균 처리 시간을 **3.33s → 1.33s (-60%)** 로 단축한 상태였습니다(현 아키텍처는 자료수집을 쟁점 갱신에 종속시켜 호출 자체를 줄이는 방향으로 전환).

### 실측에서 드러난 한계 (정직히 표기)

1. **silence 트리거 0건**: 배치 업로드는 utterance가 한꺼번에 주입되므로 "발화 간 침묵" 감지 로직이 작동하지 않음. 이 트리거는 **실시간 WebSocket 경로에서만 의미**. 배치는 STT·분석 검증용.

3. **쟁점 구조화 확률적 변동**: 동일 입력·동일 모델로 2회 run 시 positions·consensus·decision 결과가 조금씩 달라짐. gemma4:e2b temperature 특성. **고정 seed + JSON 스키마 강제(pydantic)** 로 개선 예정.

## 6. 의식한 트레이드오프 — 프로젝트 정체성 3개

### ① 클라우드 LLM vs 로컬 → **로컬 (요구사항)**
속도·품질 모두 클라우드가 유리. 그러나 회의 내용이 민감 데이터라 **진입 요건 자체가 로컬**. 이 제약이 이후 설계 방향 전체를 결정했습니다.

### ② LLM 가속 vs 호출 구조 재설계 → **호출 구조 재설계 (핵심 통찰)**
4bit 양자화·mlx 가속을 먼저 시도 → 540회 호출 앞에서 가속만으로는 실시간 불가 → 방향을 **"언제 안 부를지"** 로 전환. **이 결정 하나가 이 프로젝트의 정체성**입니다.

### ③ 정밀도 vs 재현율 → **재현율 우선**
트리거/자료수집 모두 **사용자가 무시하는 건 쉽고 놓치는 건 치명적**. 불용어 ~50개로 오탐을 억제하되 키워드는 확장적으로 유지. 자료 수집도 부동의 감지 시 **모든 엔티티**에 검색을 트리거 — 미검색의 손실이 검색 비용보다 큽니다.

### 채택하지 않은 선택지
- **클라우드 LLM API**: 민감 데이터라 제외.
- **파인튜닝 소형 모델**: 라벨링 데이터 부재로 후순위.
- **Word-level 스트리밍 STT**: 발화 경계 분리 품질이 핵심 → 문장 단위 유지.
- **순수 룰 엔진**: 확장성 부족 → LLM을 fallback으로 남긴 하이브리드.

## 7. 한계 & 배운 것

### 알려진 한계
| 한계 | 대응 |
|---|---|
| 동시 발화(overlap) 미처리 | 세그먼트 내 화자 분할 리팩토링 |
| 짧은 발화 화자 식별 정확도 | 적응형 임계값 + 사후 재처리 이원화 |
| 로컬 LLM 품질 제약 (gemma4:e2b) | 모델 인터페이스 분리 — 업그레이드 즉시 반영 |
| VAD 비음성 임펄스 오탐 | **Silero 통합 준비, 현재 RMS로 동작 — 품질 업그레이드 경로 확보** |

### 배운 것 (3줄)
1. **호출 횟수 설계 > 모델 속도** — 성능 최적화를 속도 문제가 아니라 호출 구조 문제로 재정의
2. **도메인 이해가 엔지니어링을 이긴다** — 회의 역학의 구조적 특성을 관찰했기에 LLM 없는 판정이 가능
3. **측정 인프라가 첫 번째 기능이다** — `_StepTimer`가 이후 모든 최적화 판단의 근거

### 다음 단계 (우선순위 순)
1. ✅ **실측 로그 분석** — 경로 A+B=98.9%, 절감률 99.2% 확인 (2026-04-24, 병렬 gather 적용 후)
2. **실녹음 재측정** — TTS → 실제 사람 녹음 데이터셋
3. **라벨링 데이터셋** (회의 3-5건) → 쟁점 구조화 F1
4. **고정 seed + JSON 스키마 강제(pydantic)** → 쟁점 구조화 재현성 개선
5. 파라미터 스윕 (500토큰, 400ms 등) — 품질 메트릭 확립 후
6. Silero VAD 통합 → 비음성 오탐 근원 제거
7. 세그먼트 재설계 → 동시 발화 처리

## 부록 A: Tier 1 예상 질문 실전 답변 스크립트

### Q1. "원칙 3개가 결국 전부 'LLM 덜 쓰기' 아닌가요?"
**A**: 공통 뿌리는 "LLM의 비용을 설계로 관리한다"입니다. 하지만 **적용 메커니즘이 다릅니다**.
- 원칙 1은 **시퀀스** — 값싼 판정을 먼저 (순서의 문제)
- 원칙 2는 **빈도** — 언제 부를지를 정보량으로 (타이밍의 문제)
- 원칙 3은 **병렬성** — UI와 분석을 분리 (체감의 문제)

이 세 축은 공존 가능하면서 서로 다른 공격면을 커버합니다.

### Q2. "540회 계산이 순차 가정 아닌가요? 병렬로 쏘면 되잖아요."
**A**: 병렬화도 함께 활용합니다 — Ollama는 `OLLAMA_NUM_PARALLEL>=2` 환경변수로 서버측 동시 처리를 활성화하고, transcript/analysis push도 `asyncio.create_task`로 분리해 UI 블로킹을 막습니다. (쟁점구조화·자료수집은 자료수집을 쟁점 갱신에 종속시켜 호출 자체를 줄이는 방향으로 바꾸면서 직렬화했습니다 — 평균 호출 횟수가 0에 가까워 병렬 이득이 사라졌기 때문.)

다만 병렬화만으로는 540회를 풀 수 없다는 게 핵심입니다:
- **KV 캐시를 요청별로 복제** → consumer Mac의 unified memory에서 STT·화자 임베딩과 경쟁하며 swap 유발
- **컴퓨트 바운드** → 단일 GPU/ANE에서 동시 inference는 선형 가속이 아님
- 복수 인스턴스도 모델 메모리 N배 점유 → 로컬 요구사항과 충돌

**즉 병렬화는 "도구"이지 "해결책"이 아닙니다.** 병렬로 쏴도 540회가 60회가 되진 않으므로, **"애초에 부를 횟수를 줄이기"** 를 주축으로 잡고 병렬화는 부가 최적화로 붙였습니다. 실측상 LLM 호출이 발화 174개 중 4회뿐이라 병렬화의 체감 효과는 거의 없고, **설계 원칙 1·2가 실제 절감 99.2%를 만들었습니다.**

### Q3. "실제로 경로 A/B 비율이 어떻게 나왔어요?"
**A**: 실측으로 **A+B = 98.9%** (토픽 감지 LLM fallback이 174 발화 중 **2회만** 발동). 12.5분 회의 1건, edge-tts 합성 오디오 기준 (`meeting_id=61`). 설계 가설 ~90%를 넘는 결과라 원칙 1의 전제가 강하게 뒷받침됐습니다. 단 **합성 음성 기준 한계가 있어 실제 사람 녹음으로 재측정**하는 것이 다음 마일스톤입니다.

### Q4. "gemma4:e2b로 한국어 쟁점 구조화 품질 나오나요?"
**A**: 품질 메트릭(F1 등)은 아직 없습니다. 대신 **구조적 실패 방지**를 설계로 다뤘습니다:
- Few-shot 프롬프트로 출력 양식 고정 (`_FEW_SHOT_EXAMPLE`)
- gemma 꼬리 마커(`<channel|>`) 대응: 첫 `{` ~ 마지막 `}` 구간만 추출 (`analysis/llm.py:128-135`)
- 파싱 실패 시 skip, 다음 배치에서 누적 재시도
- `_merge_positions()` 후처리로 positions·consensus·decision 필드 보정 (`analysis/issues.py:120-147`)

**현 상태의 솔직한 한계**: 동일 입력 2회 run 시 결과가 조금씩 달라짐 (gemma4:e2b temperature 특성). **고정 seed + JSON 스키마 강제(pydantic)는 다음 마일스톤**입니다. 프롬프트 코드를 바로 열어 보여드릴 준비는 되어 있고, 모델 교체가 용이한 인터페이스라 품질이 아쉬우면 더 큰 모델로 즉시 업그레이드 가능합니다.

### Q5. "정확도 측정은요?"
**A**: 이번 범위는 **"LLM 호출 경제성 증명 + 측정 인프라 구축"** 으로 의도적으로 한정했습니다. 이번 성과는:
① **LLM 호출 절감률 99.2% [실측]** (12.5분 회의, 522회 가정 → 4회)
② 측정 인프라 (`_StepTimer` + `analyze_meeting_log.py`)
③ 트리거 5/6 발동 검증

F1/precision/recall은 **품질 축**이라 라벨링 데이터셋이 선결 조건인 별도 프로젝트입니다. 다음 순서는 실녹음 재측정 → 라벨링 3-5건 → F1 → 파라미터 스윕. **스코프를 명확히 한 선택**이지 누락이 아닙니다.

### Q6. "400ms 기준 출처가 뭐죠? 요즘 업계는 100ms 아닌가요?"
**A**: Doherty-Thadani(1982, IBM)가 출발점이지만 그 이후로 Nielsen 등 UX 연구에서 **100ms=즉각, 1초=무리없음** 으로 수렴했습니다. STT 자체 지연이 150-300ms라, 이를 고려해 **400ms를 "첫 피드백" 목표**로 잡았습니다.

포인트는 수치가 아니라 기준의 **정의**입니다. 400ms는 "전체 분석 완료"가 아니라 **"내 말이 인식됐다는 첫 신호"** 기준이고, 원칙 3(비동기 디커플링)이 이걸 가능하게 만듭니다.

### Q7 (킬러). "당신이 이 프로젝트에서 증명한 본인만의 역량 한 가지만 말해보세요."
**A**: **속도 최적화를 "모델을 빠르게"가 아니라 "호출 구조를 바꾸기"로 재정의하는 감각**입니다. 첫 방향이 막혔을 때, 문제를 가속에서 빈도로 재정의한 순간이 이 프로젝트의 전환점이었고, 설계 원칙 3개는 전부 그 재정의의 파생물입니다.

### Q8 (킬러). "다시 만든다면 뭘 제일 먼저 바꿀래요?"
**A**: **측정부터.** `_StepTimer`를 첫날부터 넣었다면 원칙 1~3에 도달하는 속도가 최소 3배 빨랐을 겁니다. "성능 최적화는 추측이 아니라 측정에서 시작"이라는 교훈을 구조에 반영한 것이 이 프로젝트 최대의 학습입니다.

### Q9 (킬러). "LLM이 2년 뒤 10배 빨라지면 이 설계가 무효화되나요?"
**A**: **부분 무효화됩니다.** 원칙 1(계층형 판정)의 "cheap-first" 근거는 약해집니다. 하지만 원칙 2(정보량 밀도 기반)는 여전히 유효 — 밀도 없는 호출은 속도와 무관하게 낭비입니다. 원칙 3(비동기)도 UX 원리라 유지됩니다. 또한 "빨라짐"은 보통 "더 큰 맥락 처리 가능"과 함께 오므로, **호출 횟수 설계**의 중요성 자체는 유지됩니다.

### Q10 (킬러). "다른 후보는 F1=0.87을 들고 왔어요. 당신은 왜 숫자가 없죠?"
**A**: 숫자 하나는 확보해뒀습니다 — 12.5분 회의 기준 **LLM 호출 절감률 99.2%** [실측] (522회 가정 → 실호출 4회). 그 후보의 F1은 **품질** 축이고, 제 숫자는 **경제성** 축입니다. F1은 라벨링 데이터셋이 선결 조건이라 다음 마일스톤이지만, 측정 인프라는 이미 준비돼 있어 데이터만 확보하면 도달 시간은 짧습니다. 즉 **"얼마나 정확한가"가 아니라 "얼마나 경제적으로 구조화하는가"를 먼저 측정했고, 두 번째 축은 대기 중**입니다. 숫자의 부재가 아니라 축 선택입니다.
