# MeetingMind 발표 시나리오 (v2)

> 목적: 기업 채용 과제 발표
> 관점 전환: "수치 자랑" → "문제를 어떻게 진단하고, 어떤 원칙으로 설계했는가"
> 면접관이 보고 싶어하는 것: **결과**가 아니라 **사고 과정**과 **판단 근거**

---

## 발표 흐름

```
1. 한 줄 요약 — 이 프로젝트의 메타 통찰
2. 문제 재정의 — "진짜 어려운 지점"을 어떻게 찾았나
3. 질문 전환 — "LLM을 빠르게?" 가 아니라 "언제 LLM이 필요 없는가?"
4. 설계 원칙 4가지 — 모든 최적화의 뿌리
5. 원칙의 적용 — 각 모듈은 원칙의 사례
6. 측정 인프라와 정직한 한계 — 왜 magic number가 "최적값"이 아닌가
7. 의식한 트레이드오프 — 무엇을 양보했고, 왜
8. 배운 것 / 남은 질문
```

---

## 1. 한 줄 요약

> **회의 AI의 본질은 "언제 생각할지"를 판단하는 것이다.**
> LLM을 잘 쓰는 게 아니라, LLM을 **언제 쓰지 않을지** 결정하는 설계가 핵심이다.

이 한 문장을 얻기까지의 사고 과정이 이 발표의 전부입니다.

---

## 2. 문제 재정의 — "왜 이게 어려운가?"

### 2-1. 단순한 녹취 도구와 무엇이 다른가

다음 기능은 일반적인 회의 도구도 제공합니다.
- 음성 인식(STT), 화자 분리, 회의록 자동화

**MeetingMind가 풀고 싶었던 것은 그 위의 레이어**:
- "지금 누가 무엇을 주장하는지" 실시간 구조화
- "논의가 교착됐다 / 결정 없이 넘어갔다" 자동 감지
- "반대 의견이 나왔다" → 관련 자료 자동 수집

즉, 전사(transcription)가 아니라 **회의 역학(dynamics) 모델링**이 목표였습니다.

### 2-2. 이 목표가 만드는 세 가지 제약의 충돌

| 제약 | 요구 | 충돌 상대 |
|-----|-----|---------|
| **로컬 퍼스트** | 회의 내용은 민감 — 외부 API 금지 | 성능 |
| **실시간성** | 도어티 임계(400ms) 이내 반응 | 정확도 |
| **다기능** | 토픽/쟁점/트리거/자료 전부 | 자원 |

이 셋은 기본적으로 서로 긴장 관계입니다:
- 정확도를 올리려면 → LLM을 많이 쓰게 됨 → 속도/자원 한계
- 속도를 올리려면 → LLM을 줄여야 함 → 정확도 한계
- 로컬 환경은 → GPU 풍족한 클라우드에 비해 → 자원 한계

**과제의 난이도는 "어느 하나 해결"이 아니라 "세 제약의 교집합에서 동작하는 해"를 찾는 데 있었습니다.**

### 2-3. 처음에 빠졌던 함정

초기 설계에서는 "LLM 추론을 얼마나 빠르게 만들 수 있는가"에 집중했습니다.
- mlx-whisper로 STT 가속
- 4bit 양자화 벤치마크 (`benchmark_cohere_4bit.py`, `benchmark_whisper_4bit.py`)
- 모델 경량화

그런데 계산해보니:

```
30분 회의 × 발화 180개 × (토픽 LLM + 쟁점 LLM + 엔티티 LLM) = LLM 540회
발화당 2~7초 × 180회 ≈ 6~21분의 누적 지연
→ "아무리 LLM이 빨라져도 실시간이 불가능"
```

**이 순간 문제 진단이 바뀌었습니다.**
→ "LLM을 빠르게 만드는 게 아니라, LLM을 덜 호출하는 게 해법이다."

---

## 3. 질문 전환 — 이 프로젝트 전체를 관통하는 통찰

### 잘못된 질문
> "어떻게 LLM을 더 빠르게 만들 수 있을까?"

→ 모델 경량화, 양자화, 추론 엔진 교체 같은 **엔지니어링 공학적 답**으로 이어짐
→ 하지만 본질적 한계를 못 넘음 (로컬 환경에서는 한계가 엄존)

### 올바른 질문
> **"회의의 어떤 순간에, LLM 없이 판단이 가능한가?"**

→ **도메인 구조 분석 → 대부분의 결정은 결정론적 패턴**이라는 관찰로 이어짐
→ "언제 검색해야 하는가?" "언제 토픽이 바뀌었는가?" 같은 결정은 **언어 이해가 아니라 패턴 감지** 문제

이 질문 전환이 설계 전체를 바꿨습니다. 이후의 4가지 원칙은 전부 이 질문의 변주입니다.

---

## 4. 네 가지 설계 원칙

### 원칙 1. Cheap-First 필터링 (계층형 필터)

> 비싼 판단을 하기 전에, 값싼 판단을 먼저 통과시킨다.

**유사 패턴**: 컴퓨터 아키텍처의 캐시 계층, DB 쿼리 플래너의 인덱스 스캔 우선
**적용 예**: 토픽 감지 3단계 필터 (키워드 → 복합키워드 → LLM)

**이 원칙이 주는 효과**:
- 최악의 경우(fallback LLM) 지연은 그대로지만, 평균 지연이 극적으로 낮아짐
- LLM이 "판정 주체"에서 "예외 처리기"로 역할이 바뀜

---

### 원칙 2. 정보량 기반 트리거 (이산 → 연속)

> "몇 번 일어났는가"가 아니라 "얼마나 쌓였는가"로 판단한다.

**동기**: "발화 10개마다 호출"이라는 초기 규칙은 **정보량의 편차를 무시**함
- "네", "맞아요", "그렇죠" × 10개 → 거의 정보 없음
- 긴 논증 2개 → 실질적 내용 풍부

**해결**: 발화 수 → **토큰(문자) 누적량** 기준으로 배치

**이 원칙이 주는 효과**:
- "정보 밀도"가 낮은 구간에서 LLM 자원 절약
- 밀도가 높아지면 자연스럽게 분석 빈도 증가 → 적응형 동작

**정직한 한계**: 500자라는 임계값은 경험적 출발점일 뿐, 최적값은 실사용 데이터로 튜닝해야 함. 이 구조가 **튜닝을 가능하게 만드는 것**이 본질적 기여.

---

### 원칙 3. 도메인 구조 활용 (언어 이해 ≠ 패턴 감지)

> 회의 역학의 많은 부분은 언어 이해가 아니라 **패턴 감지** 문제다.

**관찰**:
- "합의"는 "동의합니다 / 그렇게 하죠 / 좋습니다" 등 **제한된 표현**으로 수렴
- "논의 교착"은 **같은 단어 반복**이라는 외형적 신호로 드러남
- "결정 없이 넘어감"은 **토픽 전환 + 결정 필드 비어있음** 이라는 상태 결합
- "시간 초과"는 **단순한 시계 비교**

이것들은 전부 LLM 없이 **키워드·카운트·타임스탬프**로 감지 가능합니다.

**적용 예**: 6종 개입 트리거 전부 LLM 0회, 판정 시간 < 10ms

**이 원칙이 주는 효과**:
- 판정 지연이 사실상 0 → 도어티 임계(400ms) 안에 여유 시간 확보
- 트리거가 LLM 비용에 묶이지 않으므로, **자주 실행해도 부담 없음**

**의식한 약점**: 키워드 기반은 "생각하지 못한 표현"에 취약함 → 정기적으로 오탐/누탐 케이스를 사전에 추가해야 함. 이 자체가 제품 운영 과제.

---

### 원칙 4. 사용자 체감 ≠ 시스템 완료 (비동기 디커플링)

> 도어티 임계는 "모든 처리 완료"가 아니라 "첫 피드백"의 기준이다.

**핵심 통찰**: 사용자는 "전체 분석 완료"를 기다리지 않음. "내 말이 인식되었다"는 신호만 0.4초 안에 오면 체감 반응성이 확보됨.

**설계**: 파이프라인을 두 개의 push로 분리
- `transcript` push → STT 완료 즉시 (100~300ms) → **도어티 임계 안쪽**
- `analysis` push → 백그라운드 Task로 LLM 호출 → 완료 시점에 추가 push

**이 원칙이 주는 효과**:
- 경로 C/D (LLM 호출 필요)도 **체감상 느리지 않음**
- 분석 품질을 타협하지 않으면서 UX를 확보 — "속도 vs 정확도"의 이분법을 탈출

**구현 상세 (발표 시 지나가며 언급)**: `asyncio.create_task`로 분석 경로 분리, WebSocket 메시지 타입으로 transcript/analysis 구분.

---

## 5. 원칙의 적용 — 각 모듈은 원칙의 사례

| 모듈 | 적용 원칙 | 핵심 설계 선택 |
|-----|---------|-------------|
| 토픽 감지 | 1 (cheap-first) + 3 (도메인 구조) | 3단계 필터. LLM은 **fallback**일 뿐 |
| 쟁점 구조화 | 2 (정보량 기반) | 토큰 누적 배치 + delta 반영 |
| 개입 트리거 | 3 (도메인 구조) | 6종 전부 키워드/카운트/타임스탬프 |
| 자료 수집 | 1 (cheap-first) + 3 (도메인 구조) | 부동의 감지를 **게이트**로 사용 |
| 파이프라인 전체 | 4 (비동기 디커플링) | transcript/analysis 분리 push |

**자료 수집에 주목할 점**:
- "언제 검색할지"를 LLM에게 묻지 않음
- `positions >= 2 && consensus == null` 이라는 **상태 조합**으로 결정
- 이건 회의의 구조적 특성("반대 의견이 나왔을 때만 사실 확인이 필요")을 코드로 옮긴 것

## 6. 측정 인프라와 정직한 한계 인식

### 6-1. 무엇을 측정했는가

`_StepTimer`로 모든 발화의 단계별 시간을 로깅:

```
[파이프라인 발화#7] 총 1.23초 |
  DB저장 0.01s (1%) | STT교정 0.00s (0%) |
  토픽감지 0.02s (2%) | 트리거감지 0.01s (1%) |
  쟁점구조화 0.00s (0%) | 자료수집 1.19s (97%)
```

→ **병목이 어디인지** 가시적으로 드러남
→ 최적화 전후 비교, 경로별 분포 분석 가능

### 6-2. 무엇을 아직 측정하지 못했는가 (정직하게)

| 미측정 항목 | 왜 안 했는가 | 필요한 것 |
|----------|-----------|---------|
| 쟁점 구조화 품질 (F1 등) | Ground truth 데이터셋 부재 | 실제 회의 3-5건 수동 라벨링 |
| 토픽 감지 정밀도/재현율 | 시나리오 시뮬레이션 데이터 필요 | 경계 케이스가 포함된 스크립트 회의 |
| 트리거 오탐률 | 장시간 실사용 로그 필요 | 베타 테스트 |
| 파라미터 최적값 (500토큰, 800ms 침묵 등) | 품질 메트릭이 선결 과제 | 위의 품질 지표 확립 후 스윕 |

### 6-3. 약점을 강점으로 재프레이밍

> "500토큰이 최적인지 증명하지 못했다"는 약점이 아니라,
> **측정 인프라를 먼저 깔아서, 향후 튜닝을 데이터 기반으로 가능하게 만든 것**이 설계 기여입니다.

파라미터 하나를 실험실에서 최적화하는 것보다,
**실환경에서 파라미터를 계속 튜닝할 수 있는 시스템**을 만드는 것이 더 실용적입니다.

## 7. 의식한 트레이드오프

면접에서 **"왜 A 대신 B를 선택했는가?"** 질문에 대비한 선택 근거:

### 7-1. 재현율 vs 정밀도

| 결정 | 선택 | 근거 |
|-----|-----|-----|
| 트리거 오탐 vs 누탐 | 누탐 최소화 (불용어 34개로 오탐 억제, 하지만 키워드 확장적) | **사용자가 무시하는 건 쉽지만, 놓치는 건 치명적** |
| 자료 수집 광범위 vs 정확 | 부동의 감지 시 **모든 엔티티**에 검색 트리거 | 검색 비용 대비 미검색의 손실이 더 큼 |

### 7-2. 속도 vs 정확도

| 결정 | 선택 | 근거 |
|-----|-----|-----|
| 토픽 감지 Stage 2 (LLM 없이 즉시 판정) | 속도 우선 | 오판 시 사용자가 수동 수정 가능 (UI로 복구 가능한 결정) |
| STT 교정 배치 크기 | 5발화 유지 | 더 줄이면 품질 저하, 더 늘리면 지연 체감 |

### 7-3. 단순함 vs 일반성

| 결정 | 선택 | 근거 |
|-----|-----|-----|
| VAD: Silero(DL) vs RMS(규칙) | RMS 우선 (Silero 코드 준비만) | Apple Silicon에서 지연 이슈, 초기엔 단순한 것으로 동작 우선 |
| 화자 식별: 실시간 vs 사후 재처리 | 실시간 (정확도 희생) | 회의 중 "누가 말했는지" 즉시 보는 게 UX 핵심 |

### 7-4. 고려했지만 채택하지 않은 선택지 (rejected alternatives)

- **클라우드 LLM API**: 속도·품질 모두 유리. 하지만 **회의 내용이 민감 데이터**라 제외.
- **파인튜닝한 소형 모델**: 좋은 방향이지만 **라벨링 데이터 부재**로 후순위.
- **Word-level 실시간 스트리밍 STT**: 지연은 더 짧아지지만, **발화 경계 분리 품질이 핵심**이라 문장 단위 확정 방식 유지.
- **회의 규칙 기반 결정 엔진(순수 룰)**: 확장성 부족. LLM을 fallback으로 남겨둔 하이브리드가 장기적으로 유리.

## 8. 한계 & 배운 것

### 8-1. 현재 시스템의 알려진 한계

| 한계 | 원인 | 대응 계획 |
|-----|-----|---------|
| 동시 발화(overlap) 미처리 | 세그먼트 단위 화자 식별 설계 | 세그먼트 내 화자 분할 리팩토링 |
| 짧은 발화 화자 식별 정확도 | 임베딩 품질이 발화 길이에 민감 | 적응형 임계값 + 사후 재처리 이원화 |
| 로컬 LLM 품질 제약 (gemma3:4b) | 모델 용량 한계 | 모델 교체 용이한 인터페이스 설계 — 업그레이드 시 품질 즉시 반영 |
| VAD 비음성 임펄스 오탐 | RMS는 에너지만 봄 | Silero DL VAD 통합 (코드 준비됨) |

### 8-2. 이 프로젝트에서 배운 것 (면접관에게 하고 싶은 말)

1. **"성능 최적화 ≠ 빠르게 만들기"**
   → 호출 빈도 자체를 설계로 줄이는 것이 더 큰 효과

2. **"도메인 이해가 엔지니어링을 이긴다"**
   → 회의 역학의 구조적 특성을 관찰했기에 LLM 없는 판정이 가능했음

3. **"측정 인프라가 첫 번째 기능"**
   → `_StepTimer`를 초기에 넣은 것이 모든 이후 최적화의 근거가 됨

4. **"사용자 체감은 시스템 완료와 다르다"**
   → 비동기 디커플링이 "속도 vs 정확도" 이분법을 해소

5. **"정직한 한계 인식이 신뢰를 만든다"**
   → 모든 magic number를 "최적값"으로 포장하기보다, "현재 가설 + 튜닝 가능한 구조"로 설명하는 것이 엔지니어링적 성숙

### 8-3. 더 시간이 주어지면 다음에 할 것

우선순위 순:
1. **품질 메트릭 확립** (쟁점 구조화 F1, 토픽 감지 precision/recall)
2. **파라미터 스윕** (위 메트릭을 objective로)
3. **실사용 데이터 기반 오탐/누탐 케이스 수집 → 키워드/불용어 사전 보강**
4. **Silero VAD 통합으로 오탐 근원 제거**
5. **동시 발화 처리를 위한 세그먼트 재설계**

## 부록 A: 발표 시 면접관이 던질 만한 질문 대비

| Q | A 요지 |
|---|-------|
| "500토큰이 최적인가요?" | 최적 아님. 경험적 출발점. 품질 메트릭 확립 후 스윕 예정. 구조가 **튜닝 가능하게 만든 것**이 본질 기여. |
| "왜 LLM을 더 빠르게 만들지 않았나요?" | 시도했음 (4bit 벤치마크 존재). 하지만 계산해보니 호출 **횟수**가 본질 병목. 방향을 바꿈. |
| "클라우드 API 쓰면 되지 않나요?" | 회의 내용이 민감 데이터 → 로컬 제약은 요구사항. 이 제약이 설계 방향을 결정. |
| "키워드 방식은 확장성이 없지 않나요?" | 네, 그래서 LLM을 **fallback으로 유지**. 하이브리드 구조. 또한 부족 케이스를 로그에서 수집해 사전에 추가. |
| "정확도 측정은요?" | 아직 정량 측정 안 됨. 이게 다음 순위 과제. 측정 인프라(`_StepTimer`, 로그 파이프라인)는 구축됨. |
| "실시간 화자 식별이 부정확하면 쟁점 구조화도 틀리지 않나요?" | 맞음. 그래서 화자별 Position 병합은 최대 5개로 제한, UI에서 수정 가능하게 설계. **알고 있는 약점**. |
| "왜 비동기로 나눴나요?" | 도어티 임계는 "전체 완료"가 아닌 "첫 피드백" 기준. transcript만 0.4초 안에 보내면 체감 반응성 확보. |

## 부록 B: 시연 시나리오 (기존 유지 — 4번 섹션 보조용)

### "분기 마케팅 예산 회의"

```
[00:00] 회의 시작 → 자동 "회의 시작" 토픽 생성

[00:30] A: "이번 분기 마케팅 예산을 20% 늘려야 합니다"
        → 쟁점 구조화 시작 (Position A: 증액)
        → 프로그레스 바: 48/500 (원칙 2 시각화)

[01:00] B: "작년에 늘렸는데 ROI가 3%밖에 안 나왔잖아요"
        → Position B 추가 (부동의 감지)
        → **자료 수집 트리거** (원칙 1+3 — 부동의 게이트)

[03:00] B: "수치를 봐야 하는데..."
        → info_needed 트리거 (원칙 3 — LLM 0회)

[04:00] A: "다음 안건으로..."
        → 토픽 전환 감지 (Stage 2 — LLM 0회)
        → no_decision 경고 (상태 조합)

[10:00] 동일 단어 반복 → loop 트리거

[15:00] 자동 요약 생성
```

**시연에서 강조할 것**: 각 순간에 "LLM이 호출되었는지, 안 되었는지"를 명시적으로 보여주기 — 원칙이 실제로 동작함을 시각화.
