# MeetingMind 발표 시나리오 (초안)

> 목적: 프로젝트 발표용 시나리오 구조 + 수치화 가능 포인트 정리
> 상태: 초안 — 포인트 선정 및 측정 방법 논의용

---

## 발표 흐름 (안)

```
1. 도입 — 문제 정의 & 데모
2. 아키텍처 — 전체 파이프라인 개요
3. 핵심 최적화 — "0.4초 안에 인터랙션" (도어티 임계)
4. 지능형 분석 — 쟁점 구조화 / 개입 트리거 / 자료 수집
5. 측정 결과 — Before/After 수치 비교
6. 시연 — 실시간 회의 시나리오
7. 한계 & 향후 계획
```

---

## 1. 도입 — 문제 정의

### 기존 회의의 문제
- 회의록 작성에 집중하면 참여도 하락
- 논점이 반복되거나 결정 없이 넘어가는 안건
- "그 데이터 확인해봐야 하는데" → 확인 안 되고 넘어감
- 사후 회의록 정리 시간 소모

### MeetingMind가 해결하는 것
- 실시간 음성 인식 + 화자 식별 → 자동 회의록
- 안건별 쟁점 구조화 → 누가 무슨 입장인지 실시간 정리
- 논의 교착/반복/시간초과 → 자동 개입 알림
- 반대 의견 발생 시 → 관련 자료 자동 검색

## 2. 아키텍처 개요

```
마이크 ─→ VAD ─→ STT(mlx-whisper) ─→ 화자식별(sherpa-onnx)
                                         │
                                         ▼
                                   Pipeline.on_utterance()
                                         │
                     ┌───────────────────┼───────────────────┐
                     ▼                   ▼                   ▼
               토픽 감지           트리거 감지          쟁점 구조화
             (3단계 필터)        (6종 키워드)        (토큰 기반 배치)
                                                          │
                                                          ▼
                                                   [부동의 감지 시]
                                                     자료 수집
                                                  (캐싱 + 웹검색)
                                         │
                                         ▼
                                   WebSocket push
                                   (실시간 UI 반영)
```

### 발표 포인트
- **로컬 퍼스트**: 외부 API 없이 Apple Silicon(MLX) 위에서 전부 동작
- **모듈 독립성**: 각 분석 모듈이 Pipeline에 플러그인 형태로 결합
- **3가지 입력 경로**: 실시간 마이크 / 파일 업로드 / 텍스트 시뮬레이션

## 3. 핵심 최적화 — 도어티 임계 (Doherty Threshold)

> **도어티 임계(Doherty Threshold)**: 시스템 응답이 0.4초(400ms) 이내이면
> 사용자가 "즉각적"으로 느끼고 생산성이 급격히 향상된다는 UX 원칙.
> MeetingMind에서는 "발화 완료 → UI에 첫 피드백"까지의 시간을 이 기준으로 최적화.

### 3-1. STT 레이턴시 최적화

| 항목 | 내용 | 수치화 방법 |
|------|------|-------------|
| **MLX 가속** | mlx-whisper large-v3-turbo 사용, Apple Silicon GPU/ANE 활용 | faster-whisper(CPU) vs mlx-whisper 동일 오디오 STT 시간 비교 |
| **모델 워밍업** | 서버 시작 시 더미 오디오로 첫 추론 지연 제거 | 워밍업 전/후 첫 발화 처리 시간 비교 |
| **적응형 VAD** | RMS 기반 에너지 VAD + 초기 2초 자동 캘리브레이션 | VAD 오탐률(비음성→발화 오인) 측정, 환경별 비교 |
| **침묵 감지 튜닝** | 800ms 침묵 → 발화 종료 판정 | 문장 분리 정확도 vs 응답 지연 트레이드오프 |

**측정 방법**:
```
[STT 레이턴시] = 침묵 감지 완료 시점 → Utterance 객체 생성 시점
목표: < 300ms (VAD 판정 + STT 추론)
```

### 3-2. 파이프라인 단계별 레이턴시

이미 `_StepTimer`가 모든 단계 시간을 측정 중. 로그 예시:
```
[파이프라인 발화#7] 총 1.23초 | DB저장: 0.01s (1%) | STT교정: 0.00s (0%)
| 토픽감지: 0.02s (2%) | 트리거감지: 0.01s (1%) | 쟁점구조화: 0.00s (0%) | 자료수집: 1.19s (97%)
```

**최적화 전후 비교 포인트**:

| 단계 | 최적화 전 | 최적화 후 | 개선 내용 |
|------|-----------|-----------|-----------|
| 쟁점 구조화 | 발화 5개마다 LLM 호출 | 토큰 500자 누적 시 호출 | 짧은 발화("네", "맞아요") 10개 ≠ 긴 논증 2개, 실질적 정보량 기준 배치 |
| 자료 수집 | 매 발화마다 LLM+웹검색 | 부동의 감지 시에만 검색 | 불필요한 검색 제거, 쟁점 없는 발화에서 ~1초 절약 |
| 자료 수집 (캐싱) | 동일 키워드 반복 검색 | query 캐시로 중복 스킵 | 같은 엔티티 N번 언급 시 검색 1회로 제한 |
| STT 교정 | 매 배치 LLM 호출 | 5발화 배치 유지 (도메인 컨텍스트 반영) | 교정 품질과 빈도 균형 |

### 3-3. "0.4초" 달성 시나리오 분석

**발화 → UI 반영까지의 경로별 시간**:

```
[경로 A] 일반 발화 (쟁점 미축적, 합의 중)
  VAD 침묵 감지: ~0ms (이미 완료)
  STT (mlx-whisper): 100~300ms
  DB 저장: ~10ms
  토픽 감지 (stage 1 키워드 미매칭): ~1ms
  트리거 감지 (키워드 스캔): ~1ms
  쟁점 구조화 (토큰 미달, 스킵): ~0ms
  자료 수집 (부동의 없음, 스킵): ~0ms
  WebSocket push: ~1ms
  ─────────────────────────────────
  합계: ~120~320ms ✅ 0.4초 이내

[경로 B] 토픽 전환 발화 (키워드 2개 매칭)
  STT: 100~300ms
  토픽 감지 (stage 2 즉시 판정): ~1ms
  나머지: ~15ms
  ─────────────────────────────────
  합계: ~120~320ms ✅ 0.4초 이내

[경로 C] 토픽 전환 + LLM 필요 (키워드 1개만)
  STT: 100~300ms
  토픽 감지 (stage 3 LLM 호출): 500~2000ms
  ─────────────────────────────────
  합계: 600~2300ms ❌ 초과 → 하지만 "transcript" 먼저 push

[경로 D] 쟁점 + 자료검색 동시 트리거
  STT: 100~300ms
  쟁점 구조화 LLM: 1000~3000ms
  자료 수집 LLM+웹: 1000~5000ms
  ─────────────────────────────────
  합계: 2000~8000ms ❌ 초과 → 비동기 처리로 대응
```

**핵심 주장**: 
- 경로 A/B가 전체 발화의 **~80% 이상** (대부분의 발화는 LLM 호출 불필요)
- 경로 C/D는 **비동기 아키텍처**로 대응:
  - `transcript` (발화 텍스트)는 STT 완료 즉시 push → **0.4초 이내**
  - `analysis` (쟁점/자료)는 백그라운드 Task로 실행 → 완료 시 별도 push
  - 사용자는 발화 텍스트를 즉시 보고, 분석 결과는 점진적으로 갱신됨

**수치화 방법**:
- 시나리오별 발화 100개 이상 시뮬레이션
- 경로 분포 측정: A/B/C/D 각 몇 %
- `transcript` push까지의 p50/p95/p99 레이턴시

## 4. 지능형 분석 기능

### 4-1. 쟁점 구조화 (Issue Structuring)

**발표 포인트**:
- 발화 수 기반 → 토큰(문자 수) 기반 배치로 최적화
- "네", "맞습니다" 같은 짧은 발화는 정보량이 적으므로 토큰 기준이 더 합리적
- 토큰 진행 상태를 UI 프로그레스 바로 시각화 (`구조화 245/500`)
- LLM 호출 시 기존 구조 + 최근 발화만 전달 (점진적 delta 반영)
- 화자별 Position 자동 병합 (최대 5개)

**수치화 방법**:
```
[최적화 전] 발화 10개마다 LLM 호출
  - 10분 회의(~60발화) → LLM ~6회 호출
  - 짧은 발화 10개 = "네 알겠습니다 맞아요..." → 정보 없는데 LLM 호출

[최적화 후] 500자 누적마다 LLM 호출
  - 동일 회의 → LLM ~3-4회 호출 (실질적 정보 기준)
  - LLM 호출 횟수 ~40% 감소 (발화 길이 분포에 따라 변동)
```

### 4-2. 개입 트리거 (Intervention Triggers)

**6종 트리거 — 모두 키워드/규칙 기반 (LLM 비용 0)**:

| 트리거 | 조건 | 레벨 | 의미 |
|--------|------|------|------|
| consensus | "그렇게 하죠", "동의합니다" 등 | INFO | 합의 감지 → 기록 |
| info_needed | "확인해봐야", "자료가 있나" 등 | INFO | 정보 필요 → 검색 트리거 가능 |
| no_decision | 토픽 전환 시 이전 토픽 decision=null | WARNING | 결정 없이 넘어감 경고 |
| loop | 최근 10발화 내 동일 단어 3회+ 반복 | WARNING | 논의 교착 감지 |
| silence | 침묵 5초 이상 | INFO | 진행 멈춤 감지 |
| time_over | 토픽 시작 후 10분 경과 | WARNING | 시간 초과 경고 |

**발표 포인트**:
- LLM 없이 키워드 + 패턴으로 즉시 판정 → **도어티 임계 충족**
- 불용어 사전(34개) + 안건별 중복 방지로 오탐 억제
- 시간 표시 포함 (`HH:MM:SS`)으로 회의 진행 맥락 유지

### 4-3. 자료 자동 수집 (Reference Collection) — 최적화

**최적화 전 (문제)**:
```
매 발화 → LLM 엔티티 추출 → 엔티티별 웹검색
- "PG사" 10번 언급 → 10번 검색
- "네 알겠습니다" → LLM 호출 → entities=[] → 낭비
- 캐싱 없음 → 동일 쿼리 반복
```

**최적화 후**:
```
부동의 감지 시에만 → LLM 엔티티 추출 → 캐시 확인 → 미검색 엔티티만 웹검색
- 합의 중이면 검색 자체 안 함
- 동일 쿼리 캐시 히트 → 검색 스킵
```

**수치화 방법**:
```
[시나리오: 30분 회의, 발화 180개, 엔티티 포함 발화 ~60개]

최적화 전:
  - LLM 엔티티 추출: 180회
  - 웹검색: ~60회 (엔티티 추출된 발화)
  - 동일 키워드 중복 검색: ~30회

최적화 후:
  - 부동의 구간 발화: ~30개 (전체의 ~17%)
  - LLM 엔티티 추출: 30회 (83% 감소)
  - 웹검색: ~15회 (캐시 히트 제외)
  - 동일 키워드 중복: 0회

→ LLM 호출 83% 감소, 웹검색 75% 감소
```

### 4-4. 토픽 감지 (3단계 필터)

**발표 포인트 — LLM 호출 최소화 설계**:
```
Stage 1: 키워드 매칭 (비용 0) → ~90% 발화 여기서 탈락
Stage 2: 키워드 2개+ 동시 매칭 (비용 0) → LLM 없이 즉시 토픽 생성
Stage 3: LLM 판정 (비용 1) → Stage 1 통과한 ~10%만 대상
```

**수치화 방법**:
- 시나리오 시뮬레이션에서 각 Stage 도달 비율 측정
- Stage 3 LLM 호출률 = 전체 발화 대비 몇 %

## 5. 측정 계획 — Before/After 수치 비교

### 5-1. 레이턴시 측정 (이미 인프라 있음)

`_StepTimer`가 매 발화의 단계별 시간을 로그로 출력 중.
시나리오 스크립트(`scripts/scenario_*.py`) 실행 → 로그 파싱 → 집계.

| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| transcript push 지연 | STT 완료 → WS send 시점 | p95 < 400ms |
| 파이프라인 총 시간 (LLM 미호출) | `_StepTimer` 총합 (경로 A/B) | p95 < 200ms |
| 파이프라인 총 시간 (LLM 호출 포함) | `_StepTimer` 총합 (경로 C/D) | p95 < 5s |
| LLM 호출 횟수/회의 | 로그 카운트 | 최적화 전 대비 40%+ 감소 |
| 웹검색 횟수/회의 | 로그 카운트 | 최적화 전 대비 70%+ 감소 |

### 5-2. 품질 측정

| 지표 | 측정 방법 |
|------|-----------|
| 쟁점 구조화 정확도 | 수동 평가 — 실제 회의 녹음 3건의 쟁점을 사람이 정리한 것과 비교 |
| 토픽 감지 정밀도/재현율 | 시나리오 회의에서 토픽 전환 Ground Truth 대비 F1 |
| 트리거 오탐률 | 시나리오 회의에서 개입 알림 적절성 수동 평가 |
| STT 교정 효과 | 교정 전/후 도메인 용어 정확도 비교 |

### 5-3. 비용 효율 측정

| 지표 | 최적화 전 | 최적화 후 |
|------|-----------|-----------|
| 30분 회의 LLM 호출 총 횟수 | ~250회 | ~50회 (예상) |
| 30분 회의 웹검색 횟수 | ~60회 | ~15회 (예상) |
| 발화당 평균 파이프라인 시간 | ~2.5초 | ~0.3초 (LLM 미호출 시) |

> 위 수치는 시나리오 시뮬레이션으로 실측 필요. 현재는 예상치.

## 6. 시연 시나리오 (안)

### 시나리오: "분기 마케팅 예산 회의"

```
[00:00] 회의 시작 — 자동 "회의 시작" 토픽 생성
[00:30] Speaker A: "이번 분기 마케팅 예산을 20% 늘려야 한다고 봅니다"
        → 쟁점 구조화 시작 (Position A: 예산 증액)
        → 프로그레스 바: 구조화 48/500

[01:00] Speaker B: "작년 같은 시기에 예산을 늘렸는데 ROI가 3%밖에 안 나왔잖아요"
        → Position B 추가 (반대 입장)
        → 부동의 감지 → 웹검색 트리거
        → "마케팅 예산 ROI" 관련 자료 자동 수집

[02:00] Speaker A: "그건 채널 믹스가 잘못됐기 때문이고..."
        → 쟁점 구조화 delta 업데이트 (A 입장에 argument 추가)

[03:00] Speaker B: "어쨌든 수치를 봐야 하는데..."
        → info_needed 트리거 발동 (Alert 카드 표시)

[04:00] Speaker A: "확인해봐야 할 것 같고, 다음 안건으로 넘어가죠"
        → 토픽 전환 감지 (키워드 "다음 안건")
        → no_decision 경고 (이전 토픽 결정 없음)

[10:00] 반복 논의 발생
        → loop 트리거 발동 ("반복 키워드: 예산, 채널, ROI")

[15:00] 회의 종료
        → 자동 요약 생성 (안건별 정리, 결정 사항, 액션 아이템)
```

### 시연에서 보여줄 것
1. 발화 입력 즉시 텍스트 표시 (도어티 임계 충족)
2. 프로그레스 바로 구조화 진행 상태 시각화
3. 부동의 시점에 자동 자료 검색

## 7. 발표 강조 포인트 정리

### "이렇게 최적화했다" — 기술적 차별점

| # | 최적화 | 기존 문제 | 해결 방법 | 효과 |
|---|--------|-----------|-----------|------|
| 1 | STT 레이턴시 | CPU 기반 Whisper 추론 지연 | MLX 가속 (Apple Silicon GPU/ANE) + 모델 워밍업 | STT 추론 시간 단축 |
| 2 | LLM 호출 최소화 (토픽) | 매 발화 LLM 판단 가능 | 3단계 필터 (키워드→키워드2개→LLM) | ~90% 발화에서 LLM 호출 0 |
| 3 | LLM 호출 최소화 (쟁점) | 발화 수 기준 배치 (짧은 발화 낭비) | 토큰(문자 수) 기준 배치 | LLM 호출 ~40% 감소 |
| 4 | 웹검색 최적화 | 매 발화 검색 + 캐싱 없음 | 부동의 시에만 검색 + 쿼리 캐싱 | 웹검색 ~75% 감소 |
| 5 | 비동기 아키텍처 | 분석 완료까지 UI 블로킹 | transcript 즉시 push + analysis 백그라운드 | 체감 응답 0.4초 이내 |
| 6 | 트리거 경량화 | 개입 판단에 LLM 사용 시 지연 | 키워드 + 패턴 기반 (LLM 0회) | 트리거 판정 < 10ms |

### "이런 기능을 넣었다" — 제품적 차별점

| 기능 | 왜 필요한가 | 기존 회의 도구와의 차이 |
|------|-------------|-------------------------|
| 쟁점 구조화 | 회의 중 "지금 누가 뭘 주장하는지" 실시간 파악 | 단순 전사(transcription)를 넘어 논점 그래프 제공 |
| 6종 개입 트리거 | 퍼실리테이터 없이도 회의 품질 유지 | 합의/교착/시간초과 등 회의 역학 자동 감지 |
| 부동의 기반 자료 수집 | 의견 대립 시 사실 확인이 필요 | "언제 검색할지"를 회의 맥락에서 판단 |
| 화자별 입장 추적 | 누가 찬성/반대인지 시각적 정리 | 단순 발화 목록이 아닌 구조화된 논점 뷰 |
| STT 교정 | 회의 도메인 맥락 반영한 정확도 향상 | 범용 STT 결과를 회의 컨텍스트로 후처리 |
| 프로그레스 바 | 시스템 상태 투명성 (구조화까지 얼마나 남았는지) | 사용자가 시스템 동작을 이해할 수 있는 UX |

## 부록: 수치 측정 실행 계획

### A. _StepTimer 로그 수집

기존 시나리오 스크립트 활용:
```bash
# 시나리오 실행 + 로그 파일 저장
python scripts/scenario_a.py 2>&1 | tee logs/scenario_a.log

# 로그에서 파이프라인 타이밍 추출
grep "파이프라인 발화" logs/scenario_a.log
```

### B. LLM 호출 횟수 카운트

```bash
# 쟁점 구조화 LLM 호출
grep "ask_json" logs/scenario_a.log | wc -l

# 엔티티 추출 LLM 호출
grep "엔티티 추출" logs/scenario_a.log | wc -l
```

### C. Before/After 비교

1. 최적화 전 코드로 시나리오 실행 → 로그 저장
2. 최적화 후 코드로 동일 시나리오 실행 → 로그 저장
3. 지표 비교:
   - 총 LLM 호출 횟수
   - 총 웹검색 횟수
   - 발화당 평균/p95 파이프라인 시간
   - transcript push 지연 p95

### D. 도어티 임계 달성 증명

transcript push 시점 측정:
```
STT 완료 시점 (whisper_stt.py 로그) → WS send 시점 (websocket.py 로그)
차이 = transcript 레이턴시
```
